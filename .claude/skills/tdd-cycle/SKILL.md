# Skill: tdd-cycle

Проводит реализацию задач из `tasks.md` по TDD-циклу (ADR 0009).

## Предусловие

Есть заполненная спека: `docs/specs/NNN-*/{spec.md,plan.md,tasks.md}`. Если нет —
сначала скилл `write-spec`.

## Цикл для каждой задачи tasks.md

1. **Red.** Написать тест, описывающий поведение из задачи. Запустить — убедиться,
   что падает (компиляция или ассерт). Падающий тест подтверждает, что он
   реально что-то проверяет.
2. **Green.** Минимальный код, чтобы тест прошёл. Не больше нужного.
3. **Refactor.** Почистить при зелёных тестах. **Тесты не переписывать под код.**

Тест и код — в одном инкременте (ADR 0003).

## Параллельные треки (если tasks.md их размечает)

Проверить `tasks.md` на раздел «Треки и параллельность». Если его нет —
выполнять весь чеклист последовательно (см. «Правила»), остальное этого
раздела не касается.

Если раздел есть:

1. **Волна 0** (обычно только контракты) — выполнить самому, последовательно,
   закоммитить в рабочую ветку.
2. Для следующей волны с несколькими треками — запустить по одному вызову
   `Agent` на трек, **все вызовы в одном сообщении** (реальный параллелизм),
   `isolation: "worktree"`. Промпт агента — самодостаточный: путь к
   `spec.md`/`plan.md`/`tasks.md`, список задач именно этого трека (номера +
   текст), явное перечисление файлов трека из таблицы. Каждый агент проходит
   свои задачи по обычному red→green→refactor и коммитит в своей ветке.
   Если его задачи требуют контрактов/SQL — он сам гоняет `make generate`/
   `make sqlc` у себя в worktree (сгенерированный код не в git, между
   worktree не переносится).
3. Дождаться завершения всех агентов волны. Смержить их ветки в рабочую по
   очереди (`git merge <branch>`) — треки дизъюнктны по файлам, конфликтов
   быть не должно. Если конфликт всё же возник — треки размечены неверно;
   разобраться и поправить `tasks.md`, не подавлять `--no-verify` и не решать
   силовым мерджем. После мержа — `make generate && make sqlc` один раз в
   основной рабочей копии (перегенерация с учётом всех смерженных контрактов).
4. Продолжить со следующей join-задачи `tasks.md` строго последовательно —
   до следующей волны с треками (если есть), затем повторить шаги 2–3.

## Уровень теста по типу изменения (ADR 0009)

| Изменение                     | Тест первым                                     |
| ----------------------------- | ----------------------------------------------- |
| `pkg/` утилита                | `pkg/*_test.go`                                 |
| `service/` юзкейс             | `service/*_test.go` с fake-репо (`testutil/`)   |
| Новый/изменённый RPC          | `api/*_test.go` (httptest + Connect-клиент)     |
| Сложный SQL/миграция          | Интеграционный с БД (по необходимости)          |
| BFF route / web-логика        | `*.test.ts` рядом (Vitest)                      |

Для нового RPC обычно оба: юнит на юзкейс (все доменные ошибки) + e2e на маппинг
в `connect.Code`.

## Команды

- Server: `go test ./...` (в CI `-race`), `go build ./...`.
- Web: `pnpm test` (Vitest) + **обязательно** `pnpm exec tsc --noEmit` после
  правок тестов с protobuf-моками (Vitest не ловит `TS2322/TS2345`).
- Всё вместе: `make test-all`.
- При изменении контрактов/SQL: `make generate` / `make sqlc` перед тестами.

## Правила

- Порядок задач в `tasks.md` соблюдать (контракты → server снизу вверх → web →
  проверка). Если `tasks.md` размечает треки — см. «Параллельные треки» выше;
  внутри трека и в join-задачах порядок по-прежнему обязателен.
- Fake-репозитории — в `modules/*/testutil/`, имя `Fake<Name>`, не `Mock`.
- Обязательно покрывать: счастливый путь, все доменные ошибки, граничные случаи
  (пустой email, короткий пароль, невалидный/истёкший токен).
- По завершении — отметить задачи в `tasks.md`, обновить статусы спеки и индекс
  в `docs/specs/README.md`.
