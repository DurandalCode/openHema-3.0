# ADR 0007: RBAC (user/admin) + бутстрап первого админа через env

- Статус: принято
- Дата: 2026-07-06

## Контекст

До этого инкремента в системе существовала единственная роль — «пользователь».
Концепции `role`/`admin` не было ни в БД, ни в proto, ни в JWT-claims, ни в
домене. Все RPC `AuthService` (Register/Login/Refresh/Me) были публичными
(ни один интерсептор аутентификации не работал). ADR 0005 явно откладывал
RBAC и admin-зону в будущий инкремент.

Инкремент вводит две роли (`user`, `admin`) и полный набор admin-операций:
админы создаются только админами. Ключевой вопрос — **как завести первого
админа** в опенсорс-проекте, не зашивая секреты в репозиторий или миграции.

## Решение

### 1. Модель роли

- **БД**: колонка `auth.users.role TEXT NOT NULL DEFAULT 'user'` с
  `CHECK (role IN ('user','admin'))` + partial index `WHERE role='admin'`.
  Строка (не enum PG-тип) — проще миграций и совместимо с sqlc.
- **Proto**: `enum Role { ROLE_UNSPECIFIED=0; ROLE_USER=1; ROLE_ADMIN=2 }` +
  поле `Role role = 5` в `User`. Типобезопасно на уровне контракта; маппинг
  в/из доменной строки — в api-слое (`toProtoRole`).
- **Domain**: `type Role string` с константами `RoleUser`/`RoleAdmin`.
- **JWT**: claim `rol` несёт роль в access-токене. Refresh-токен роль не
  несёт — при обновлении роль берётся из БД, чтобы учесть её изменение
  без перевыпуска всех токенов.

### 2. Доверяем JWT-claim на admin-RPC

На admin-операциях роль проверяется по JWT-claim без дополнительного запроса
к БД. Access-TTL = 15 минут — для пет-проекта с единственным админом это
достаточный компромисс между строгостью и простотой. Понижение админа
учитётся при следующем логине/refresh (роль перечитывается из БД). Если позже
потребуется мгновенный отзыв — это будет отдельный ADR по token revocation
(уже отмечен как будущий в ADR 0001).

### 3. Отдельный AdminService + per-handler RequireAdmin

Admin-операции вынесены в отдельный `AdminService` (proto `admin.proto`), а не
примешаны к `AuthService`. Доступ ограничен интерсептором `RequireAdmin`,
который накладывается **per-handler** на `AdminService` в `module.Register`,
а не глобально. Глобальный `Auth`-интерсептор валидирует Bearer access-токен
для всех RPC, кроме публичных (`Register/Login/Refresh` — белый список
процедур). Таким образом, admin-хендлер получает и `Auth` (через base-opts),
и `RequireAdmin` (через admin-opts).

### 4. Guards DemoteUser

`DemoteUser` отклоняет операцию (`ErrForbidden` → `CodePermissionDenied`) в
двух случаях:
- **Нельзя понизить себя** (`userID == callerID`) — админ не должен иметь
  возможности лишить себя последних привилегий ошибочным кликом.
- **Нельзя понизить последнего админа** (`CountAdmins <= 1`) — система не
  должна остаться без ни одного администратора.

### 5. Бутстрап первого админа — env-переменные при старте

Механизм: env `BOOTSTRAP_ADMIN_EMAIL` + `BOOTSTRAP_ADMIN_PASSWORD` +
`BOOTSTRAP_ADMIN_DISPLAY_NAME`. При старте сервера (`platform.New`, до приёма
запросов) вызывается `auth.Bootstrap` → `service.BootstrapAdmin`:

1. Если email или password пусты → skip (бутстрап не настроен).
2. `CountAdmins > 0` → skip (админ уже есть, идемпотентно).
3. Иначе — хешировать пароль (argon2id) и `CreateUser(role=admin)`.
4. При `ErrUserExists` (админ с таким email уже есть) → skip.

Логируется результат: `created` / `skipped` / `failed`.

**Почему не альтернативы:**
- **Миграция с зашитым админом** — нельзя: опенсорс, секреты в репо.
- **CLI-подкоманда** — рассмотрена, но для пет-проекта с потенциально 1
  админом env-бутстрап проще (ноль ручных шагов, авто-восстановление при
  пустой БД). CLI остаётся как будущее расширение (promote/list/demote из
  терминала), если понадобится.
- **«Первый зарегистрировавшийся = админ»** — отклонено из-за security smell:
  на свежем опенсорс-деплое атакующий может первым зарегистрироваться и
  получить админку.
- **Setup-токен** — усложнение (хранение/expire/single-use) избыточно для
  единственного аккаунта.

**Race при параллельном старте двух инстансов:** оба могут пройти `CountAdmins==0`
и попытаться вставить. `UNIQUE(email)` гарантирует, что успешна будет одна
вставка, вторая получит `ErrUserExists` → graceful skip. Согласованность
сохранена.

**Рекомендация по эксплуатации:** после первого входа админа ротировать
пароль и убрать `BOOTSTRAP_ADMIN_*` из окружения. Бутстрап идемпотентно
пропустится.

### 6. Guard admin-зоны на вебе — server-component layout, не middleware

ADR 0005 предполагал `middleware.ts` guard для admin-зоны. На практике
edge-`middleware` в Next.js не может читать JWT из httpOnly-cookie (токены
только в `src/lib/session`, серверные) и ходить в gRPC. Поэтому реальная
проверка роли выполняется в `app/(admin)/layout.tsx` (server component):
`getCurrentUser()` → не admin → `redirect("/")`. Это осознанное отклонение
от ADR 0005 с обоснованием здесь.

## Последствия

- Добавлена миграция `00001` (правка в рамках раннего дева — разовое
  исключение; далее новые инкременты отдельными миграциями).
- `pkg/connectutil` получил `Auth` и `RequireAdmin` интерсепторы +
  context-keys (`CallerID`/`CallerRole`).
- JWT-сигнатура `Issue` изменилась: `Issue(userID)` → `Issue(userID, role)`.
  Все вызывающие стороны обновлены.
- Веб-клиент получает `role` через `User` proto-сообщение (BFF прокидывает
  как есть через `userToJson`); admin-зона рендерится только для админов.
- Будущее: token revocation (ADR), CLI для admin-операций, гейтинг публичной
  регистрации — отдельные инкременты.
