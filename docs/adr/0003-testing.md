# ADR 0003: Стратегия тестирования

- Статус: принято
- Дата: 2026-07-06

## Контекст

Проект разрабатывается нейросетями (AI-first). Нужна стратегия тестирования,
которая ловит регрессии на каждом инкременте, но не требует тяжёлой инфраструктуры
локально. Бóльшая часть логики — серверная (Go), поэтому фокус тестирования там.

## Решение

### Обязательное правило

Каждый инкремент содержит тесты. PR без тестов не принимается.

### Пирамида (server, Go)

1. **Юнит-тесты** — чистая логика без внешних зависимостей:
   - `pkg/*` — утилиты (crypto, jwt, config).
   - `modules/*/service/` — юзкейсы с **fake-репозиторием** (in-memory
     `domain.Repository`), без реальной БД.

2. **E2E тесты ручек** — полный путь через Connect: proto-запрос → хендлер →
   service → маппинг ошибок → proto-ответ. Через `httptest` + сгенерированный
   Connect-клиент. Не требуют реальной БД (fake-репозиторий).

3. **Интеграционные с БД** (по необходимости) — реальный PostgreSQL для
   миграций и sqlc-запросов. Добавляются позже при появлении сложных запросов.

### Test doubles

Fake-репозитории в `modules/*/testutil/` — ручные in-memory реализации портов.
Не генерируемые моки. Используются и service-, и handler-тестами — один test
double на порт.

### Клиент (web)

Vitest для чистой логики (маппинг ошибок, сериализация). Скриншотные тесты
намеренно не используем. UI-тесты — по мере появления реальных компонентов.

## Обоснование

- Fake-репозитории делают тесты быстрыми и не зависящими от Docker/Postgres.
- E2E через `httptest` + Connect-клиент тестирует реальный транспортный путь
  (proto-сериализация, маппинг ошибок в `connect.Code`) без сетевой инфраструктуры.
- Тестирование доменных ошибок обязательно — это контракт между модулями.

## Последствия

- Каждый модуль имеет `testutil/` с fake-реализациями портов.
- Тесты запускаются локально без Docker: `go test ./...`.
- `make test-all` — единая команда для CI.
