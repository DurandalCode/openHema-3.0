# ADR 0011: Event-Driven Design (event sourcing) для флоу заявок

- Статус: принято
- Дата: 2026-07-10

## Контекст

Спека 0005 (флоу подачи заявки бойца) вводит первый в проекте bounded context,
где источник истины — не строка в таблице, а **неизменяемый журнал доменных
событий**: заявка реконструируется как агрегат сверткой своего потока событий.
Причина — не техническая мода, а доменное требование: заявка проходит
растянутый во времени процесс (подача → оплата → подтверждение → регистрация,
либо отзыв), и организатору нужно видеть **как** она пришла в текущее
состояние (кто и когда подал/оплатил/подтвердил/отозвал) для разбора спорных
и денежных ситуаций (FR-9, FR-10 спеки 0005).

До этого инкремента в проекте не было ни event store, ни конвенций по
оптимистичной конкуренции, ни соглашений об именовании событий. Нужно
зафиксировать эти решения один раз, чтобы последующие событийные фичи (если
появятся) их переиспользовали, а не изобретали заново.

## Решение

### 1. Единица консистентности — агрегат

Один поток событий = один агрегат (`aggregate_id`). В флоу заявок — это одна
заявка (`applicant_user_id` × `nomination_id` в момент подачи, спека 0005,
решение 4). Команды идут по циклу:

```
load stream (события агрегата) → rebuild (fold → текущее состояние + версия)
  → decide (доменная команда: разрешён ли переход, кто инициатор)
  → append (новое событие, ожидаемая версия)
```

Никакого глобального event log на все агрегаты сразу — партиционирование по
`aggregate_id` естественным образом даёт параллелизм между разными заявками.

### 2. Event store — append-only таблица, не шина

Событие — факт, случившийся один раз; UPDATE/DELETE над журналом не
выполняются никогда. Схема таблицы (пример — модуль `application`):

- `aggregate_id`, `version` (1-based, порядок в потоке), `event_type`,
  `payload` (jsonb, полезная нагрузка), `actor_id`, `occurred_at`.
- `UNIQUE(aggregate_id, version)` — одновременно (а) физический порядок
  событий в потоке и (б) механизм оптимистичной конкуренции (см. п.3).

Это **не** шина сообщений и не brokered messaging (Kafka/NATS и т.п.) — на
этом этапе проекта отдельная событийная инфраструктура избыточна. Каждый
event-sourced модуль ведёт собственный журнал в своей PG-схеме (ADR 0002).
Если позже понадобится публикация событий **между** модулями — это отдельное
расширение (см. п.5), не меняющее внутренний журнал.

### 3. Оптимистичная конкуренция по версии потока

Команда загружает поток, знает его текущую версию `v`, и пытается вставить
новое событие с `version = v+1`. `UNIQUE(aggregate_id, version)` гарантирует:
если две команды одновременно решили действовать над одной и той же версией
`v`, вставится только одна — вторая получит нарушение уникальности.

**Политика на конфликт:** один прозрачный повтор (reload → redecide → reappend);
если конфликтует и повторно — отдать `ErrConcurrency` наружу
(`connect.CodeAborted`). Слепой ретрай-цикл не делаем — конфликт на втором
проходе means состояние действительно гонится, и клиент должен попробовать
снова осознанно.

### 4. Инлайн-проекция (read-model) в той же транзакции

Реконструкция агрегата сверткой всего потока при каждом чтении корректна, но
дорога для списков (все заявки номинации, «мои заявки», счётчики) и не может
выразить некоторые инварианты на самом журнале — например, «нет двух активных
заявок одного пользователя в одну номинацию» (FR-12 спеки 0005): это
ограничение на *текущее состояние*, а не на отдельное событие.

Поэтому каждый event-sourced модуль ведёт **инлайн-проекцию** — таблицу
текущего состояния агрегата (`application_current` и т.п.), обновляемую
**атомарно, в одной транзакции** с `INSERT` события (append event + upsert
projection = одна transaction). Проекция:

- обслуживает дешёвые чтения (списки, счётчики, SSR) — NFR-1 спеки 0005;
- несёт инварианты текущего состояния через обычные PG-констрейнты (в
  флоу заявок — partial unique index на активные состояния);
- **выводима**: в принципе может быть перестроена из журнала с нуля (переигрывание
  потока), журнал остаётся единственным источником истины.

Снапшоты агрегата (оптимизация `Rebuild` для длинных потоков) — вне скоупа
этого ADR; при необходимости — отдельное решение поверх той же модели.

### 5. Кроссдоменные эффекты — пока только факт в журнале

Терминальное событие может по смыслу принадлежать другому домену (в спеке
0005 — `FighterRegistered` формально относится к домену бойцов, которого пока
нет). Правило: пока принимающий домен не существует, событие **только
фиксируется** в журнале своего модуля — никакой синхронной записи в чужую
схему и никакой самодельной шины «на всякий случай».

Когда принимающий домен появится, кроссдоменная доставка идёт **через API
модуля** (ADR 0002), не через прямой доступ к чужой БД. Конкретный механизм
(синхронный вызов при append, polling журнала, outbox-паттерн) выбирается той
будущей фичей и не предрешается здесь — важно, что смена механизма доставки не
требует переписывать уже записанную историю (совместимость вперёд, NFR-2).

### 6. Наименование событий

Событие называется в прошедшем времени и формулирует свершившийся факт
предметной области, а не техническое действие: `ApplicationSubmitted`, не
`CreateApplication`; `PaymentConfirmed`, не `UpdateStatus`. Тип события —
строка (не PG enum) для простоты миграций, набор значений валидируется на
уровне приложения и CHECK-констрейнтом.

## Область действия

Эти конвенции обязательны для **event-sourced** модулей (где домен явно
требует историю переходов как источник истины). Они не отменяют и не заменяют
обычный CRUD-подход остальных модулей (`nomination`, `tournament`) — там
текущее состояние продолжает храниться напрямую, без журнала событий. Решение,
каким должен быть новый модуль (CRUD или event-sourced), принимается на этапе
`spec.md`/`plan.md` конкретной фичи (ADR 0008) — как domain-требование, не
default.

## Последствия

- Первый event-sourced модуль по этим правилам — `application` (спека 0005).
- Каждый event-sourced модуль обязан иметь: append-only таблицу событий с
  `UNIQUE(aggregate_id, version)`, инлайн-проекцию, обновляемую атомарно с
  событием, и integration-тест, подтверждающий, что конфликт версии реально
  ловится на реальной БД (testcontainers, ADR 0010).
- Кроссдоменная публикация событий между модулями остаётся нерешённой до
  первой фичи, которой она реально понадобится (см. п.5) — не проектируем
  событийную шину заранее.
- Снапшоты, replay-инструменты, повторная сборка проекции из журнала — future
  work, не блокируют текущий инкремент.
