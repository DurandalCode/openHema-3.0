# ADR 0004: Не хранить сгенерированный код в репозитории

- Статус: принято
- Дата: 2026-07-06

## Контекст

В проекте есть два вида сгенерированного кода:
- `server/gen/`, `web/src/gen/` — из `.proto` через `buf generate`.
- `server/modules/*/repo/sqlc/` — из `.sql` через `sqlc generate`.

Раньше такие файлы часто коммитят в репо. Минусы: шум в диффах, конфликты
merge, риск рассинхрона (забыли перегенерить). Плюсы: онбординг без установки
инструментов, агенты читают gen-код без генерации.

## Решение

**Не хранить сгенерированный код в репо.** Добавить в `.gitignore`:
- `server/gen/`
- `web/src/gen/`
- `server/modules/*/repo/sqlc/`

Генерация выполняется:
- **Локально** — после `git clone`/`git pull`: `make generate && make sqlc`.
- **В CI** — каждый server/web job генерирует перед билдом/тестами.

## Обоснование

- Единственный источник истины — `.proto` и `.sql`. Вывод не хранится.
- Нет merge-конфликтов в gen-файлах.
- Нет проблемы «устаревшего gen-кода» — он всегда свежий после генерации.
- CI гарантирует, что источник генерируется (если `make generate` падает —
  пайплайн красный).

## Последствия

- **Онбординг**: каждый клонер должен установить Go (1.26+) и выполнить
  `make generate && make sqlc` перед `go build`/`pnpm build`. Документируем в
  корневом `AGENTS.md` (раздел «Онбординг»).
- **AI-агенты**: перед чтением gen-кода нужно его сгенерировать. В инструкциях
  агентам указывать `make generate && make sqlc` как первый шаг после checkout.
- **CI**: каждый server/web job добавляет шаг генерации (~5–15 сек с кэшем Go).
- `go build`/`go test`/`tsc`/`pnpm build` упадут без предварительной генерации —
  это ожидаемо и ловится локально и в CI.
