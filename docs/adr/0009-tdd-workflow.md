# ADR 0009: TDD Workflow

- Статус: принято
- Дата: 2026-07-07
- Инкремент: 2026-07-15 — параллельное выполнение независимых треков
  `tasks.md` (новый подраздел «Параллельное выполнение независимых треков»,
  правки «Связь с SDD» и «Последствия»).

## Контекст

ADR 0003 зафиксировал **стратегию** тестирования (пирамида, fake-репозитории,
e2e через httptest+Connect, Vitest) и правило «каждый инкремент содержит тесты».
Но он не задаёт **порядок** работы. Для AI-first разработки порядок критичен:
если агент пишет код первым, тест часто подгоняется под уже написанное и
перестаёт быть независимой проверкой поведения. Нужен явный TDD-цикл, связанный
с артефактом `tasks.md` из SDD (ADR 0008).

## Решение

### Обязательный цикл: red → green → refactor

Для каждой задачи из `tasks.md`:

1. **Red** — сначала тест, описывающий желаемое поведение. Он падает
   (компиляция/ассерт) — это подтверждает, что тест действительно что-то
   проверяет.
2. **Green** — минимальный код, чтобы тест прошёл. Не больше, чем нужно.
3. **Refactor** — чистка при зелёных тестах. Тесты не переписываем под код.

Тест и код идут в одном инкременте/коммите (правило ADR 0003 сохраняется).

### Какой уровень пирамиды для какого изменения

| Изменение                              | Тест первым                                            |
| -------------------------------------- | ------------------------------------------------------ |
| Утилита в `pkg/`                       | Юнит-тест `pkg/*_test.go`                              |
| Юзкейс в `service/`                    | Юнит с fake-репо (`testutil/`), `service/*_test.go`    |
| Новый/изменённый RPC                   | E2E ручки `api/*_test.go` (httptest + Connect-клиент)  |
| Сложный SQL/миграция                   | Интеграционный с БД (по необходимости, ADR 0003)       |
| BFF route handler / чистая web-логика  | Vitest `*.test.ts` рядом с кодом                       |

Для нового RPC обычно нужны оба уровня: юнит на юзкейс (все доменные ошибки) +
e2e на маппинг ошибок в `connect.Code`.

### Связь с SDD

`tasks.md` (ADR 0008) — это упорядоченный список TDD-шагов. Каждая задача
явно называет слой/файл и подразумевает пару «тест → код». Порядок задач
отражает порядок red-green-refactor **внутри одного трека** (см. ниже) — по
умолчанию весь чеклист один трек, выполняемый строго последовательно.

### Параллельное выполнение независимых треков (инкремент 2026-07-15)

Не всякий `tasks.md` обязан быть одной последовательной цепочкой. Если часть
задач не трогает файлов, которые трогает другая часть («трек»), эти части
можно выполнять одновременно — отдельными агентами, каждый в изолированном
`git worktree` (инструмент `Agent`, `isolation: "worktree"`, несколько
вызовов в одном сообщении = параллельный запуск).

- **Критерий трека — дизъюнктность файлов, а не «логическая независимость».**
  Два куска работы могут быть независимы по смыслу, но если оба правят один
  файл (например, один и тот же `service.go`) — это один трек: параллельный
  мерж дал бы конфликт. Дизъюнктность файлов — необходимое и достаточное
  условие для параллели (мерж по построению бесконфликтный).
- **Треки группируются в волны.** Волна стартует только после того, как все
  треки предыдущей волны смержены в рабочую ветку. Задачи, которым нужны
  файлы нескольких треков (platform-wiring, интеграционные тесты, финальная
  verification) — отдельная join-волна после мержа.
- **Сгенерированный код не переносится между worktree.** `gen/`,
  `*/repo/sqlc/` не в git (ADR 0004) — каждый трек сам гоняет
  `make generate`/`make sqlc` у себя в worktree после того, как нужные ему
  контракты (proto/sql) уже смержены в базу волны. После мержа всех треков
  волны координатор перегенерирует ещё раз в основной рабочей копии перед
  join-задачами.
- **Разбиение на треки фиксируется в `tasks.md`** — таблица «волна / трек /
  задачи / файлы / зависимость», см. `_templates/tasks.md` и пример
  `docs/specs/0010-pool-bouts/tasks.md`. Явных дизъюнктных кусков нет — раздел
  не нужен, чеклист остаётся одним треком (это по-прежнему дефолт: не любая
  фича достаточно велика, чтобы параллелить, и не стоит городить треки ради
  двух-трёх задач).

### Web-специфика (protobuf-моки)

Vitest транспилирует без проверки типов и **не ловит** несовместимость
plain-объектов с branded generated-типами (`$typeName`). Правило: protobuf-моки
строятся через `create(Schema, partial)` из `@bufbuild/protobuf`, и после правок
таких тестов локально обязателен `pnpm exec tsc --noEmit` (CI это проверяет
отдельным job). Подробности — в `docs/conventions.md`.

## Обоснование

- Тест-первым сохраняет тест как независимую спецификацию поведения, а не
  зеркало реализации — особенно важно, когда код пишет агент.
- Явная таблица «изменение → уровень теста» убирает догадки, какой тест писать.
- Связка с `tasks.md` делает TDD не абстрактным правилом, а конкретным
  пошаговым чеклистом.

## Последствия

- `tasks.md` каждой спеки структурирован как последовательность red-green шагов
  (по умолчанию один трек; при явно дизъюнктных кусках — несколько треков по
  волнам, см. выше).
- Скилл `tdd-cycle` проводит агента по циклу и гоняет `make test` /
  `make test-web` + `tsc --noEmit`; при размеченных треках — параллелит их
  через `Agent`/`isolation: worktree` и сериализует join-задачи после мержа.
- Дополняет, а не заменяет ADR 0003: стратегия — там, порядок — здесь.
