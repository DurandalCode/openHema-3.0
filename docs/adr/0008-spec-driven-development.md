# ADR 0008: Spec-Driven Development

- Статус: принято
- Дата: 2026-07-07

## Контекст

Базовый код (auth, RBAC, UI-каркас) готов. Дальше — сложные фичи, которые
трогают одновременно `proto`, несколько server-модулей и web-слои. Проект
AI-first: значительную часть кода пишут агенты. Без явной спецификации агент
начинает кодить от первого правдоподобного толкования задачи, а доменные
допущения всплывают уже в коде — их дорого переигрывать.

Нужен лёгкий процесс, который отделяет **что и зачем** от **как** и заставляет
проговорить открытые вопросы до генерации кода — но без тяжёлого внешнего
тулинга.

## Решение

Каждая нетривиальная фича проходит цикл **spec → plan → tasks → code**. Три
markdown-артефакта на фичу, живут централизованно.

### Где живут спеки

```
docs/specs/
  README.md                       индекс + краткое описание процесса
  _templates/{spec,plan,tasks}.md шаблоны (источник истины формата)
  NNN-<kebab-feature>/
    spec.md                       ЧТО и ЗАЧЕМ (без технологий)
    plan.md                       КАК (proto, модули, слои, БД, тесты)
    tasks.md                      упорядоченный TDD-чеклист
```

Нумерация `NNN-<kebab-feature>` — сквозная, параллельно нумерации ADR
(узнаваемо для агентов). Спеки лежат **отдельно от кода**: фича почти всегда
кросс-слойная (server + web), централизованное место не привязывает её к одному
из них.

### Три артефакта

1. **`spec.md`** — user-facing. Проблема, акторы (роли `user`/`admin` из
   RBAC), пользовательские сценарии, требования, критерии приёмки в форме
   Given/When/Then. Явный раздел `[NEEDS CLARIFICATION]` для открытых вопросов.
   **Без** API, схем БД, технологий — только предметная область.
2. **`plan.md`** — технический план, привязанный к архитектуре: изменения
   контрактов в `proto/hema/v1/*`, затрагиваемые server-модули и их слои
   (`api/service/domain/repo/migrations`), PG-схема (при новом bounded
   context), затрагиваемые web-слои (FSD + BFF), стратегия тестирования по
   пирамиде ADR 0003.
3. **`tasks.md`** — упорядоченный чеклист под TDD (см. ADR 0009): каждая задача
   привязана к слою/файлу и порядку red → green → refactor.

### Когда спека обязательна

- Новый server-модуль (bounded context) или новая web-фича.
- Новый RPC с нетривиальной доменной логикой.
- Изменение, затрагивающее контракты `proto` и несколько слоёв.

Мелкие правки (рефактор, фикс, косметика UI, правка текста) спеки не требуют —
для них достаточно обычного инкремента с тестами.

### Связь с ADR

Спека описывает **конкретную фичу**; ADR фиксирует **сквозное решение**
(паттерн, технологию, границу). Если в ходе `plan.md` рождается новое сквозное
решение — оно выносится в отдельный ADR, а `plan.md` на него ссылается.

## Обоснование

- Разделение «что» и «как» ловит доменные недопонимания до генерации кода —
  самая дешёвая точка исправления для AI-first разработки.
- `[NEEDS CLARIFICATION]` делает неопределённость явной вместо молчаливых
  допущений агента.
- Централизованные спеки естественно покрывают кросс-слойные фичи и дают
  агенту единую точку входа в задачу.
- Лёгкий markdown-процесс без внешнего тулинга не добавляет зависимостей и
  соответствует документация-driven природе проекта.

## Последствия

- Фича начинается с `docs/specs/NNN-*/` (отражено в корневом `AGENTS.md` и DoD).
- Шаблоны в `docs/specs/_templates/` — источник истины формата; правки формата
  идут туда.
- Скилл `write-spec` и команда `/spec` автоматизируют создание спеки из идеи.
- Секция событий в `plan.md` пока placeholder — наполнится с ADR по
  Event-Driven Design (следующий инкремент, вместе с первой событийной фичей).
