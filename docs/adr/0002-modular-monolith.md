# ADR 0002: Модульный монолит с границами под микросервисы

- Статус: принято
- Дата: 2026-07-06

## Контекст

Сервер начинается как монолит, но потенциально должен разбиваться на
микросервисы. Нужно заложить границы так, чтобы вынос модуля был механическим,
без переписывания бизнес-логики и вызывающего кода.

## Решение

Модульный монолит с жёсткими границами:

1. **Модуль = вертикальный bounded context**: `proto → api → service → domain →
   repo → migrations`. Не технический слой, а срез предметной области.
2. **Изоляция данных**: одна PostgreSQL-инстанция, **отдельная схема на модуль**
   (`auth.*`, `tournament.*`). Кросс-модульных JOIN нет; доступ к чужим данным —
   только через API модуля.
3. **Единая точка входа модуля**: `Register(mux, deps)`. `cmd/server`
   регистрирует все модули (монолит); `cmd/<module>` — один (микросервис).
4. **Межмодульное общение через Connect-интерфейсы**: in-process в монолите,
   по сети при выносе. Код вызывающего не меняется.
5. **Явные зависимости**: DI через конструкторы, wiring в `internal/platform`
   (composition root). Общий код без бизнес-логики — в `pkg/`.

## Структура

```
server/
  cmd/server/         композит всех модулей
  pkg/                config, logger, jwt, crypto, connectutil, pgxutil
  internal/platform/  composition root (сборка mux + wiring)
  modules/<name>/     module.go, api/, service/, domain/, repo/, migrations/
  gen/                proto→Go (DO NOT EDIT)
```

## Обоснование

- Схема-на-модуль режет связность на уровне данных заранее, без затрат
  отдельных БД сейчас.
- Одинаковый интерфейс `Register` и Connect-контракты делают вынос в сервис
  предсказуемым и дешёвым.

## Последствия

- Дисциплина: запрет прямых обращений к чужим схемам БД (ревью/линт).
- Небольшой оверхед на границы даже когда всё in-process.
- Вынос модуля: собрать `cmd/<module>`, переключить клиент на сетевой транспорт,
  перенести схему БД.
