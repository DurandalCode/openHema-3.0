# Спецификации фич (Spec-Driven Development)

> Процесс: см. `docs/adr/0008-spec-driven-development.md`.
> TDD-цикл: см. `docs/adr/0009-tdd-workflow.md`.

Каждая нетривиальная фича проходит цикл **spec → plan → tasks → code**.

## Структура

```
docs/specs/
  README.md                       этот файл
  _templates/                     шаблоны — источник истины формата
    spec.md                       ЧТО и ЗАЧЕМ (без технологий)
    plan.md                       КАК (proto, модули, слои, БД, тесты)
    tasks.md                      упорядоченный TDD-чеклист
  NNN-<kebab-feature>/            папка одной фичи
    spec.md
    plan.md
    tasks.md
```

Нумерация `NNN-<kebab-feature>` — сквозная, три цифры, kebab-case имя. Пример
формата — `0000-healthcheck`; реальные фичи нумеруются с `0001`.

## Процесс

1. **spec** — заполнить `spec.md` из шаблона. Проговорить проблему, акторов,
   сценарии, критерии приёмки (Given/When/Then). Всё неясное — в раздел
   `[NEEDS CLARIFICATION]`. Не переходить дальше, пока остались нерешённые
   критичные вопросы.
2. **plan** — заполнить `plan.md`: как ложится на `proto`, server-модули и их
   слои, PG-схему, web-слои (FSD + BFF), какие тесты по пирамиде (ADR 0003).
3. **tasks** — разложить `plan.md` на упорядоченный TDD-чеклист (`tasks.md`):
   каждая задача = слой/файл + пара «тест → код».
4. **code** — выполнять `tasks.md` по TDD-циклу red → green → refactor.

## Инструменты

- Скилл `write-spec` — создаёт `docs/specs/NNN-*/` из шаблонов и ведёт
  заполнение.
- Команда `/spec <краткое описание фичи>` — быстрый запуск SDD-цикла из чата.
- Скилл `tdd-cycle` — прогон задач из `tasks.md` по red-green-refactor.

## Когда спека нужна

Обязательна: новый модуль/фича, нетривиальный RPC, кросс-слойное изменение
контрактов. Не нужна: рефактор, баг-фикс, косметика — им хватает инкремента с
тестами.
