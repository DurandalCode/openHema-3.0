# Spec: Профиль турнира (single-tournament)

> Артефакт SDD (ADR 0008). Описывает **ЧТО и ЗАЧЕМ**. Без технологий, API и
> схем БД — только предметная область. Реализация — в `plan.md`.

- Статус: done
- Дата: 2026-07-07

## Проблема

В MVP система обслуживает **один турнир** (мультитурнирности нет). При этом у
турнира как у сущности есть публичная «визитка»: название, описание, дата
проведения, контакты для связи (Telegram, VK, Facebook) и эмблема. Сейчас эти
данные негде хранить и нельзя менять — на главной странице турнир представлен
заглушкой («Турниры скоро появятся»), а изменить название/описание/контакты без
правки кода невозможно.

Нужно дать администратору возможность настраивать профиль турнира из админки, а
посетителям — видеть его на главной странице.

## Акторы

- **admin** — редактирует профиль турнира (название, описание, дату, контакты,
  эмблему) из админки. Роль `ROLE_ADMIN` (RBAC уже есть).
- **гость / user** — видит профиль турнира на главной странице. Не
  аутентифицирован или аутентифицирован как обычный пользователь; только чтение.

## Пользовательские сценарии

1. Как **admin**, я хочу задать название, описание и дату проведения турнира,
   чтобы посетители понимали, что это за турнир и когда он проходит.
2. Как **admin**, я хочу указать контакты (Telegram, VK, Facebook), чтобы
   участники могли связаться с организаторами.
3. Как **admin**, я хочу загрузить/указать эмблему турнира, чтобы страница
   выглядела узнаваемо.
4. Как **admin**, я хочу отредактировать любое из полей позже, чтобы
   поддерживать информацию актуальной.
5. Как **гость**, я хочу увидеть на главной странице название, описание, дату,
   эмблему и контакты турнира, чтобы получить о нём представление.

## Требования

- FR-1: В MVP система работает с **одним активным турниром**. Модель данных при
  этом хранит турнир как полноценную сущность с идентификатором (готова к
  нескольким турнирам в будущем), но публичная главная и админка оперируют
  ровно **активным** турниром.
- FR-2: Профиль включает поля: название, описание, дата+время проведения (с
  временной зоной), список контактов (произвольные каналы), эмблема (URL).
- FR-3: admin может просмотреть текущий (активный) профиль турнира в админке.
- FR-4: admin может изменить любое поле профиля; изменения сохраняются и сразу
  видны на главной.
- FR-5: Профиль (или его часть) отображается на главной странице для всех
  посетителей без аутентификации.
- FR-6: Пустые/незаданные поля не ломают отображение (главная работает даже с
  частично заполненным профилем).
- FR-7: Редактирование профиля доступно только роли `ROLE_ADMIN`; чтение —
  публично.
- FR-8: Контакты — упорядоченный список записей, каждая с типом канала (напр.
  telegram, vk, facebook, website, email, other) и значением (строка). admin
  может добавлять, удалять и переупорядочивать записи.
- FR-9: Дата проведения хранится как момент времени с зоной; на главной
  отображается человекочитаемо. Поле опционально (турнир без назначенной даты
  допустим).
- FR-10: Эмблема хранится как URL на внешнее изображение (загрузка бинарных
  файлов — вне скоупа MVP). Поле опционально.

<Нефункциональные.>

- NFR-1: Чтение активного профиля для главной страницы — дешёвое (одна запись),
  пригодно для SSR при каждом заходе.
- NFR-2: Модель проектируется расширяемой под мультитурнирность: смена «MVP:
  один активный турнир» на «список турниров + выбор» не должна требовать
  переписывания сущности (только снятие ограничения единственности активного).

## Критерии приёмки

- **AC-1** (чтение публично)
  - Given: профиль турнира заполнен админом
  - When: гость открывает главную страницу без токена
  - Then: он видит название, описание, дату, эмблему и заданные контакты
- **AC-2** (редактирование админом)
  - Given: admin аутентифицирован и открыл раздел настроек турнира
  - When: он меняет название и описание и сохраняет
  - Then: изменения сохраняются и отражаются на главной странице
- **AC-3** (защита записи)
  - Given: обычный пользователь (не admin) или гость
  - When: он пытается изменить профиль турнира
  - Then: запрос отклоняется (`Unauthenticated`/`PermissionDenied`)
- **AC-4** (частично заполненный профиль)
  - Given: заданы только название и дата, контакты и эмблема пусты
  - When: гость открывает главную
  - Then: страница отображает название и дату, пустые блоки контактов/эмблемы не
    показываются и не ломают вёрстку
- **AC-5** (единственность активного)
  - Given: система в MVP-режиме (один активный турнир)
  - When: admin открывает настройки турнира
  - Then: он всегда редактирует один и тот же активный профиль, а не создаёт
    новые
- **AC-6** (список контактов)
  - Given: admin в настройках турнира
  - When: он добавляет контакт telegram и контакт website и сохраняет
  - Then: оба контакта сохраняются в заданном порядке и показываются на главной
    как ссылки; ранее заданные контакты не теряются
- **AC-7** (дата с зоной)
  - Given: admin задал дату и время проведения с временной зоной
  - When: гость открывает главную
  - Then: дата/время отображаются человекочитаемо

## Вне скоупа

- **Мультитурнирность** — несколько турниров, список, выбор активного. Осознанно
  откладываем; модель проектируем так, чтобы не мешать будущему расширению.
- **Фавиконка** — пользователь явно отметил как сложное и ненужное сейчас.
- Регистрация участников, сетки, расписания, судейство — отдельные фичи.
- Версионирование/история изменений профиля.

## Принятые решения

Вопросы закрыты в обсуждении со стейкхолдером (2026-07-07):

1. **Хранение полей** — явные колонки под известные поля (title, description,
   event_date, emblem_url) + отдельная таблица контактов. Спроектировано с
   прицелом на мультитурнирность (турнир — сущность с `id`).
2. **Эмблема** — URL на внешнее изображение (строка). Загрузка файлов — вне
   скоупа.
3. **Дата проведения** — момент времени с временной зоной (timestamptz),
   опционально.
4. **Контакты** — произвольный упорядоченный список записей {тип, значение}.
5. **Инициализация** — таблица `tournament` с настоящим `id`; в MVP ровно один
   помечен активным (`is_active`), миграция сидит один пустой активный турнир,
   админка редактирует активный (upsert-семантика для активного). Переход к
   мультитурнирности = снятие ограничения единственности активного, без
   переписывания сущности.
6. **Границы модуля** — новый server-модуль `tournament` (bounded context со
   своей PG-схемой, ADR 0002).
