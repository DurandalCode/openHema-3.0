# Spec: Постановка пула на арену (pool on arena)

> Артефакт SDD (ADR 0008). Описывает **ЧТО и ЗАЧЕМ**. Без технологий, API и
> схем БД — только предметная область. Реализация — в `plan.md`.

- Статус: done
- Дата: 2026-07-18

## Проблема

Есть площадки/ристалища (0008), есть раскладка бойцов номинации по пулам (0009),
и при фиксации раскладки (`draft → ready`) внутри каждого пула формируются бои —
пары «каждый с каждым» с готовым порядком проведения (0010). Следующий шаг
реального турнирного дня — **вывести пул на конкретное ристалище**: организатор у
каждой арены решает, какой пул сейчас будет на ней проходить, и ставит его туда
**целиком** (вместе с его уже сформированными боями и их порядком, 0010). Только
после этого секретарь арены сможет вести бои этого пула по очереди — но сам ход
боёв (текущий бой, результат) — **следующий инкремент** (ЕДД, «текущий бой на
арене»), не этот.

Сегодня связать пул с ареной негде: у арены есть стабильная страница-каркас
(0008, FR-9) с плейсхолдером «управление боями появится позже», а пул не знает,
на какой площадке он проходит. Нужна **постановка пула на арену** как отдельное
явное действие организатора и как публичная информация для бойца/зрителя: боец
должен видеть, **на какой площадке** его пул и что пул уже **готовится к
запуску**.

Попутно упрощаем статусную модель. В 0009 у раскладки номинации есть статус, где
`active/finished` были заглушкой «на будущие бои». Но на арену выходит
**отдельный пул**, а не вся номинация сразу: у одной номинации один пул может уже
стоять на арене и «готовиться к запуску», а другой — просто ждать. Значит,
**исполнительная** фаза (готовится/идёт/завершён) принадлежит **пулу**, а
**составительная** (готова раскладка или нет) — по-прежнему раскладке. Поэтому:

- **статус раскладки урезаем до двух** значений — `не готов` (draft) / `готов`
  (ready); заглушки `active/finished` с раскладки убираем;
- **пулы переходят в «готов» вместе с раскладкой** (фиксация — по-прежнему по
  всей номинации, 0009/0010 не ломаем);
- **исполнительный статус — на пуле**: `готовится к запуску` (на арене) и далее
  (`идёт`/`завершён` — будущее, ЕДД).

## Акторы

- **admin (оргкомитет/секретарь)** — составляет и **фиксирует раскладку**
  номинации (`draft → ready`, существующее действие 0009 — формирует бои 0010),
  на **странице арены** выбирает из готовых пулов и **ставит пул на арену**, при
  необходимости **снимает** его обратно; видит, какой пул на каждой арене и его
  бои. Роль `ROLE_ADMIN` (как в 0007/0008/0009, секретарь = admin).
- **боец / гость** — на **новом публичном экране номинации** видит пулы
  (готовой раскладки), их состав, бои (пары и порядок) и — если пул поставлен на
  арену — **на какой площадке** пул и что он **«готовится к запуску»**.
  Публичного управления нет, только чтение.
- **секретарь конкретной арены** (будущее, ЕДД) — ведёт текущий бой пула,
  стоящего на его арене. **Вне скоупа** этого инкремента (пока = admin).

## Домен

### Двухуровневая статусная модель

**Уровень раскладки номинации (составление)** — урезаем до двух значений
(с 0009 убираем заглушки `active/finished`):

```
не готов (draft)  ──▶  готов (ready)
```

- **не готов (draft)** — раскладка в работе (0009): DnD бойцов, создание/удаление
  пулов, автораспределение, сброс, undo.
- **готов (ready)** — раскладка зафиксирована (0009): состав заморожен, для
  каждого пула сформированы бои (0010). Обратимо (в `draft`), но **только если ни
  один пул не стоит на арене** (иначе исчезли бы бои, которые «готовятся к
  запуску»).

**Уровень отдельного пула (исполнение)** — статус пула:

```
не готов  ──▶  готов  ──▶  готовится к запуску  ──▶  идёт  ──▶  завершён
(draft)      (ready)      (preparing, на арене)   (active)  (finished)
```

- **не готов / готов** — пул **переходит вместе с раскладкой**: пока раскладка
  `draft`, пул «не готов»; при фиксации раскладки (`draft → ready`) все её пулы
  становятся «готов» (и получают бои, 0010).
- **готовится к запуску (preparing)** — пул **поставлен на арену**: закреплён за
  конкретной площадкой, его бои по порядку готовы к проведению. Обратимо: admin
  **снимает** пул с арены (→ «готов»).
- **идёт / завершён** — исполнение боёв пула. **Вне скоупа** (ЕДД, след.
  инкремент); значения — задел под целостность модели, наполнятся без ломки
  данных.

**В этом инкременте реализуются:** переход раскладки `draft ↔ ready` (существует в
0009, урезается enum) и переход пула `готов ↔ готовится к запуску` (постановка/
снятие с арены). Переходы пула `готовится → идёт → завершён` не реализуются (нет
вызывающего их действия).

### Постановка пула на арену

- **Пул принадлежит номинации** (0009, FR-1) и **может стоять не более чем на
  одной арене** (инвариант: один пул → максимум одна площадка). «Пока что, чтобы
  не усложнять с турами»: в будущем пул может проходить в несколько туров на
  разных аренах — но не в этом инкременте.
- **Одна арена ведёт один пул за раз** (инвариант: на активной арене в каждый
  момент стоит не более одного пула). Поставить второй пул на занятую арену
  нельзя, пока текущий не снят. Так арена остаётся однозначным «рабочим местом»
  под будущий текущий бой (ЕДД).
- Ставить на арену можно **только пул со статусом «готов»** (раскладка
  зафиксирована, бои сформированы — есть что исполнять). Пул «не готов»
  поставить нельзя.
- Постановка ставит пул **целиком, вместе с его боями и их порядком** (0010):
  арена не пересчитывает порядок, только будет исполнять готовый (ЕДД).
- Постановка **обратима**: пул можно **снять** с арены (вернуть в «готов»),
  освободив площадку. Снятие ничего не разрушает (бои пула сохраняются).
- **Раскладку с пулом на арене нельзя вернуть в `draft`** (пересобрать состав),
  пока пул не снят — сначала снять все пулы с арен, потом расфиксировать.
- На арену можно ставить пулы **разных номинаций** (арена межноминационна): в
  каждый момент на арене один пул, из любой номинации.
- Ставить пул можно только на **активную** (не архивную, 0008) арену. Пока на
  арене стоит пул, её нельзя убрать в архив (сначала снять пул).

### Публичное чтение пулов номинации (новое)

- Появляется **новый публичный экран номинации** (для бойца/гостя), где видно:
  - **пулы номинации** с их **составом** (имена/клубы бойцов — как публичный
    ростер 0007);
  - **бои** каждого пула (пары бойцов и порядок проведения, 0010);
  - если пул поставлен на арену — **имя площадки** и статус **«готовится к
    запуску»**.
- Экран показывается **для готовой раскладки** (`ready`): пулы и бои
  финальны. Пока раскладка `draft` (в составлении) — публично пулы **не видны**.
- Экран **read-only**: управление (постановка/снятие, фиксация) — только у admin
  и не на этом экране.

> Изменение к 0009/0010: там публичное чтение пулов и боёв было осознанно
> отложено. Этот инкремент его **вводит** для готовой раскладки (см. «Принятые
> решения» №4).

## Пользовательские сценарии

1. Как **admin**, я фиксирую раскладку номинации (`draft → ready`, 0009), её пулы
   становятся «готов» и внутри каждого формируются бои (0010) — пулы становятся
   доступны для постановки на арену.
2. Как **admin**, я открываю **страницу конкретной арены** и вижу список
   **готовых пулов** (в статусе «готов», ещё не стоящих ни на одной арене), чтобы
   выбрать, какой пул вывести на это ристалище.
3. Как **admin**, я **ставлю выбранный пул на арену** одним действием — и он
   вместе со своими боями и их порядком закрепляется за площадкой; арена
   показывает этот пул и его бои по порядку.
4. Как **admin**, я вижу, что **занятая арена** не предлагает поставить ещё один
   пул: пока текущий не снят, второй поставить нельзя.
5. Как **admin**, я **снимаю пул с арены** (например, ошибся ареной), пул
   возвращается в «готов» и снова доступен для постановки; площадка
   освобождается.
6. Как **admin**, я НЕ могу вернуть раскладку в `draft`, пока какой-то её пул
   стоит на арене — система просит сначала снять пулы с арен.
7. Как **боец/гость**, я открываю публичный экран своей номинации и вижу пулы
   готовой раскладки, их состав и бои, а также — если пул уже выведен — **на
   какой площадке** он проходит и что он **«готовится к запуску»**, чтобы
   понимать, куда идти.
8. Как **боец/гость**, пока раскладка номинации ещё составляется (`draft`), я
   пулы на публичном экране не вижу.

## Требования

Функциональные:

- FR-1: **Двухуровневый статус.** Раскладка номинации имеет статус `не готов
  (draft)` / `готов (ready)` — урезан с 0009 (значения `active/finished`
  убраны). Отдельный **пул** имеет статус `не готов → готов → готовится к запуску
  → идёт → завершён`; `не готов/готов` **синхронны со статусом раскладки**,
  `готовится к запуску` — исполнительная фаза пула. В этом инкременте
  реализуются переходы раскладки `draft ↔ ready` и пула `готов ↔ готовится к
  запуску`.
- FR-2: **Фиксация раскладки** (`draft → ready`, существующее действие 0009,
  FR-9): все пулы номинации становятся «готов», для каждого формируются бои (0010,
  FR-2/FR-3). Пул с < 2 бойцами — без боёв, без ошибки (0010, FR-4).
- FR-3: **Расфиксация раскладки** (`ready → draft`, 0009): бои всех пулов
  удаляются (0010, FR-5), пулы возвращаются в «не готов», состав снова
  редактируем. **Запрещена**, если **хотя бы один пул** номинации стоит на арене
  («готовится к запуску») — сначала снять пулы с арен.
- FR-4: **Составление** (DnD, создание/удаление пула, автораспределение, сброс,
  undo) — как в 0009: только когда раскладка `draft`. В `ready` раскладка
  read-only (0009, FR-11).
- FR-5: **Один пул — не более одной арены.** Пул в каждый момент закреплён не
  более чем за одной площадкой.
- FR-6: **Одна арена — один пул за раз.** На активной арене в каждый момент стоит
  не более одного пула. Постановка на занятую арену **отклоняется**.
- FR-7: **Постановка на арену** (пул `готов → готовится к запуску`): admin ставит
  пул со статусом «готов» на **активную** (не архивную) арену. Пул получает
  привязку к площадке и статус «готовится к запуску». Пул «не готов» (или уже на
  арене) поставить нельзя; постановка на архивную арену отклоняется.
- FR-8: **Снятие с арены** (пул `готовится к запуску → готов`): admin снимает пул
  с площадки; привязка к арене очищается, статус → «готов»; площадка
  освобождается. Бои пула сохраняются.
- FR-9: **Постановка/снятие — из панели арены** (страница конкретной арены, 0008
  FR-9). На странице арены admin видит: какой пул сейчас стоит (и его бои по
  порядку) либо список доступных для постановки готовых пулов.
- FR-10: **Архивация занятой арены запрещена** (расширение 0008, FR-5): нельзя
  убрать в архив арену, на которой стоит пул — сначала снять пул.
- FR-11: **Публичный экран номинации**: боец/гость видит пулы **готовой**
  раскладки (`ready`) с их составом (имена/клубы, 0007) и боями (пары + порядок,
  0010); для пула на арене — **имя площадки** и ярлык **«готовится к запуску»**.
  Пулы `draft`-раскладки публично не показываются. Экран read-only.
- FR-12: **«Готовится к запуску» = пул на арене.** Публичный ярлык «готовится к
  запуску» соответствует статусу пула «готовится к запуску» (пул закреплён за
  площадкой, бои готовы, но ещё не начаты). Начало боёв («идёт») — вне скоупа.
- FR-13: **Права.** Фиксация раскладки, постановка/снятие, составление — только
  `ROLE_ADMIN`. Публичный экран номинации (FR-11) доступен всем (чтение), как
  публичный ростер 0007.

Нефункциональные:

- NFR-1: Модель расширяема под `готовится к запуску → идёт → завершён` (текущий
  бой, результат — ЕДД) и под будущие туры (пул на нескольких аренах) без ломки
  данных.
- NFR-2: Постановка/снятие — синхронные операции, без фоновой очереди.
- NFR-3: Публичный экран номинации — дешёвое чтение, пригодное для SSR (как
  главная 0007): состав пулов + бои + текущее размещение на арене одним экраном.
- NFR-4: Инварианты «один пул ↔ одна арена» и «одна арена ↔ один пул» защищены на
  уровне данных, а не только приложения (конкурентная постановка не должна
  посадить два пула на одну арену).

## Критерии приёмки

- **AC-1** (фиксация раскладки → пулы готовы + бои)
  - Given: раскладка номинации в `draft`; пул P с бойцами A, B, C
  - When: admin фиксирует раскладку (`draft → ready`)
  - Then: раскладка `ready`; пул P — «готов»; для P сформированы бои A–B, A–C,
    B–C с порядком (0010)
- **AC-2** (расфиксация удаляет бои)
  - Given: раскладка `ready`, ни один пул не на арене
  - When: admin возвращает раскладку в `draft`
  - Then: раскладка `draft`; пулы — «не готов»; бои удалены; состав редактируем
- **AC-3** (нельзя расфиксировать, пока пул на арене)
  - Given: раскладка `ready`, пул P стоит на арене («готовится к запуску»)
  - When: admin пытается вернуть раскладку в `draft`
  - Then: операция отклонена (`FailedPrecondition`), просьба сначала снять пулы с
    арен; раскладка остаётся `ready`
- **AC-4** (постановка на арену)
  - Given: пул P «готов»; активная арена R, свободна
  - When: admin ставит P на R
  - Then: статус P = «готовится к запуску»; P закреплён за R; на странице R виден
    пул P и его бои по порядку
- **AC-5** (нельзя поставить «не готов» пул)
  - Given: пул P «не готов» (раскладка `draft`); свободная активная арена R
  - When: admin пытается поставить P на R
  - Then: операция отклонена (`FailedPrecondition`): пул не готов; состояние не
    изменилось
- **AC-6** (арена занята — второй пул нельзя)
  - Given: на арене R стоит пул P1 («готовится к запуску»); пул P2 «готов»
  - When: admin пытается поставить P2 на R
  - Then: операция отклонена (`FailedPrecondition`): арена занята; P2 остаётся
    «готов», R по-прежнему ведёт P1
- **AC-7** (пул уже на другой арене)
  - Given: пул P стоит на арене R1
  - When: admin пытается поставить P на арену R2
  - Then: операция отклонена (`FailedPrecondition`): пул уже на арене; сначала
    снять
- **AC-8** (снятие с арены освобождает площадку)
  - Given: на арене R стоит пул P
  - When: admin снимает P с R
  - Then: статус P = «готов»; привязка к R очищена; R свободна и снова принимает
    постановку; бои P сохранены
- **AC-9** (постановка только на активную арену)
  - Given: пул P «готов»; арена R в архиве
  - When: admin пытается поставить P на R
  - Then: операция отклонена; архивная арена пул не принимает
- **AC-10** (нельзя архивировать занятую арену)
  - Given: на арене R стоит пул P
  - When: admin пытается убрать R в архив
  - Then: операция отклонена (`FailedPrecondition`): на арене стоит пул; сначала
    снять
- **AC-11** (публичный экран: готовая раскладка, состав и бои)
  - Given: номинация с раскладкой `ready` (пул P: бойцы A,B,C; бои сформированы)
  - When: гость открывает публичный экран номинации
  - Then: видит пул P с составом A,B,C и его боями по порядку
- **AC-12** (публично: площадка и «готовится к запуску»)
  - Given: пул P стоит на арене «Ристалище 1»
  - When: гость открывает публичный экран номинации
  - Then: у пула P показано «Ристалище 1» и ярлык «готовится к запуску»
- **AC-13** (публично: пул готов, но не на арене)
  - Given: пул P «готов», не на арене
  - When: гость открывает публичный экран номинации
  - Then: P показан с составом и боями, **без** площадки и **без** «готовится к
    запуску»
- **AC-14** (публично: раскладка draft не видна)
  - Given: раскладка номинации в `draft`
  - When: гость открывает публичный экран номинации
  - Then: пулы не показаны (раскладка ещё составляется)
- **AC-15** (права на управление)
  - Given: обычный пользователь или гость
  - When: пытается фиксировать раскладку, поставить/снять пул с арены
  - Then: операция отклонена (`Unauthenticated` / `PermissionDenied`)

## Вне скоупа

- **Текущий бой на арене и его ход** (старт боя, результат, счёт, победитель,
  статусы пула «идёт»/«завершён») — следующий инкремент (ЕДД, «текущий бой на
  арене»). Здесь пул лишь «готовится к запуску», бои не идут.
- **Событийная модель (ЕДД) арены** — вводится отдельным ADR вместе с текущим
  боем; здесь связь синхронная (см. «События»).
- **Туровость** (пул проходит несколько туров, возможно на разных аренах) —
  пока один пул ↔ одна арена. Осознанное упрощение (запрос стейкхолдера).
- **Расписание/тайминг** постановки, очередь пулов на арену — не делаем: одна
  арена ведёт один пул за раз, очереди нет.
- **Ручной порядок боёв на арене** — арена исполняет готовый порядок пула (0010).
- **Физическое удаление** пулов/арен — как и раньше (0008/0009), не вводим.
- **Реакция на выбытие/снятие бойца** после фиксации раскладки — как в 0010:
  зафиксированный состав/бои — снапшот; полноценно решится с ЖЦ боя.

## Принятые решения

Зафиксировано со стейкхолдером (2026-07-18):

1. **Ёмкость арены — один пул за раз** (FR-6). Арена — однозначное рабочее место
   под будущий текущий бой (ЕДД); очередей пулов не вводим.
2. **Двухуровневый статус** (FR-1, уточнено 2026-07-18): статус раскладки
   **оставляем**, но урезаем до двух (`не готов`/`готов`), убирая заглушки
   `active/finished`. Пулы переходят в «готов» **вместе с раскладкой** (фиксация —
   по-прежнему по всей номинации, 0009/0010 без изменений). Исполнительный статус
   пула (`готовится к запуску`, далее `идёт`/`завершён`) — на пуле. _Отвергнут
   более радикальный вариант «убрать статус раскладки, всё на пул» — он ломал
   пер-номинационную композицию 0009 и триггер боёв 0010 без нужды._
3. **Постановка/снятие — с страницы арены** (FR-9), из готовых пулов; пул
   ставится целиком с боями (FR-7). Постановка обратима (FR-8).
4. **Публичное чтение пулов вводится** новым экраном номинации (FR-11): состав
   пулов + бои + площадка/«готовится к запуску». Разворачивает отложенное в
   0009/0010 публичное чтение — только для готовой раскладки.
5. **«Готовится к запуску» = пул на арене** (FR-12), исполнительный статус пула;
   начало боёв — вне скоупа.
6. **Один пул ↔ одна арена** (FR-5) — упрощение «без туров» по запросу
   стейкхолдера; модель расширяема под туры позже (NFR-1).

## Влияние на 0009/0010

- **0009 (пулы номинации):** статус раскладки **урезается** с четырёх до двух
  значений (`draft/ready`; `active/finished` убраны). Пер-номинационная фиксация
  состава (`draft → ready`) и все операции составления — **без изменений**.
  Добавляется: пул несёт собственный статус (`не готов/готов/готовится к
  запуску/…`) и привязку к арене; расфиксация (`ready → draft`) блокируется, если
  пул стоит на арене (FR-3).
- **0010 (бои внутри пула):** **без изменений.** Триггер формирования боёв —
  прежний переход раскладки `draft → ready` по всей номинации; логика round-robin
  и порядок сохраняются полностью.

Статус 0009/0010 в индексе `docs/specs/README.md` помечается «изменён 0011» (для
0010 — только урезание enum-статуса раскладки, поведение боёв не меняется).
