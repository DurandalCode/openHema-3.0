# Spec: Бои внутри пула (формирование пар)

> Артефакт SDD (ADR 0008). Описывает **ЧТО и ЗАЧЕМ**. Без технологий, API и
> схем БД — только предметная область. Реализация — в `plan.md`.

- Статус: ready
- Дата: 2026-07-15

## Проблема

Домен боёв (matches) — один из крупнейших в системе, поэтому реализуем его
маленькими инкрементами вдоль будущего бизнес-процесса, а не целиком.
Сегодняшний инкремент — **первое проявление боя**: только то, как бой
рождается из уже существующей раскладки пулов (спека 0009). Дальше по
маршруту — «постановка пула на арену» и затем полноценный жизненный цикл
одного боя (старт/результат/победитель); оба — отдельные будущие спеки.

Сейчас у номинации есть раскладка бойцов по пулам (0009) со статусом
`draft`/`ready`. Когда admin фиксирует раскладку (`draft → ready`), она
перестаёт меняться — но боёв за этим пока никаких не следует: непонятно, кто
с кем встретится внутри пула. Нужно, чтобы **сама фиксация раскладки
формировала пары бойцов на бой** — по одной паре на каждую уникальную пару
бойцов пула (round-robin, «каждый с каждым»), детерминированно и без ручного
участия admin.

На арену (будущий инкремент «постановка пула на арену») отправляется **пул
целиком**, а не отдельные бои по одному — а значит, **порядок, в котором бои
пула идут друг за другом, тоже определяется на этапе формирования боёв**, а не
позже: арена лишь исполняет уже готовый порядок. Порядок должен по возможности
избегать ситуации «боец вышел на бой сразу после своего предыдущего боя без
паузы» — это тоже часть этого инкремента.

Жизненный цикл отдельного боя (начало, результат, статус, площадка) в этом
инкременте **не реализуется** — только сам факт «эти два бойца встретятся» как
следствие готовности пула.

## Акторы

- **admin (секретарь)** — переводит раскладку номинации в `ready` (существующее
  действие, 0009 FR-9) и как следствие видит сформированные бои по каждому
  пулу; возвращает раскладку в `draft`, если нужно поправить состав, и видит,
  что бои при этом исчезают.
- **гость / user** — в этом инкременте бои не видят. В перспективе бойцам
  важно оперативно знать, когда и с кем их бой — публичное отображение боёв
  на экране номинации (там, где сейчас публичный состав пулов) появится
  отдельным будущим инкрементом; сейчас, как и раскладка пулов (0009), бои —
  внутренние данные организатора.

## Пользовательские сценарии

1. Как **admin**, я перевожу раскладку номинации в `ready` и хочу, чтобы для
   каждого пула автоматически сформировались пары «каждый с каждым» — без
   отдельной кнопки «сформировать бои».
2. Как **admin**, я хочу увидеть на экране пула список сформированных боёв
   (пар бойцов) этого пула после перехода в `ready`.
3. Как **admin**, если в пуле оказался всего один боец (или ни одного), я хочу,
   чтобы переход в `ready` всё равно прошёл успешно — просто без боёв для этого
   пула, без ошибки.
4. Как **admin**, я возвращаю раскладку в `draft`, чтобы поправить состав пулов
   (0009), и ожидаю, что ранее сформированные бои исчезнут вместе с фиксацией —
   они относились к раскладке, которая больше не действует.
5. Как **admin**, после правки состава я снова перевожу раскладку в `ready` и
   хочу получить **новый** набор пар, соответствующий актуальному составу
   пулов, а не старый.
6. Как **admin**, я хочу, чтобы сформированные бои пула шли в таком порядке,
   что боец **по возможности не выходит на бой сразу после своего предыдущего
   боя** — потому что пул на арену ставится целиком и бои внутри него идут
   один за другим без ручной пересборки порядка.

## Требования

Функциональные:

- FR-1: **Бой принадлежит ровно одному пулу** и связывает **двух разных**
  бойцов этого же пула. Боёв между бойцами разных пулов (в т.ч. разных
  номинаций) не бывает.
- FR-2: **Триггер формирования** — переход статуса раскладки номинации
  `draft → ready` (существующее действие, 0009 FR-9). Формирование —
  автоматическое следствие этого перехода; отдельного явного действия
  «сформировать бои» нет.
- FR-3: **Схема формирования — round-robin («каждый с каждым»)**: для каждого
  пула раскладки создаётся по одному бою на **каждую уникальную пару** бойцов
  этого пула (пул из `n` бойцов даёт `n×(n−1)/2` боёв). Каждая пара бойцов
  встречается **не более одного раза** в пределах пула за одно формирование.
- FR-3a: **Бои пула имеют порядок** (последовательность проведения) — пул
  ставится на арену целиком (будущий инкремент), бои внутри него исполняются
  друг за другом в этом порядке; арена его не пересчитывает, только исполняет.
  Порядок строится **турами**: бойцы разбиваются на туры так, что внутри
  одного тура боец участвует **не более чем в одном** бою (жёсткий инвариант,
  выполняется всегда — AC-9); итоговый порядок — бои первого тура, затем
  второго и т.д.
- FR-3b: **Эвристика минимизации стыков между турами**: на стыке между
  соседним турами (последний бой предыдущего тура → первый бой следующего)
  порядок боёв внутри туров подбирается так, чтобы по возможности избежать
  ситуации «боец выходит на бой сразу после своего предыдущего боя без
  боя-паузы между ними». Для маленьких пулов (3–4 бойца) такая смежность
  математически неизбежна на части стыков при любом порядке — эвристика
  её не убирает, там нечего минимизировать (AC-10). Для пулов от 6 бойцов
  такая смежность всегда полностью устранима — эвристика обязана её
  устранить (AC-11), а не просто уменьшить частоту.
- FR-4: **Пул с менее чем двумя бойцами** (0 или 1) — бои не формируются (0
  боёв для этого пула); переход раскладки в `ready` из-за этого **не
  блокируется** и ошибкой не считается.
- FR-5: **Возврат раскладки из `ready` в `draft`** (существующее действие,
  0009 FR-9) **удаляет все ранее сформированные бои всех пулов** этой
  номинации. Бои относятся к зафиксированной раскладке; как только раскладка
  снова стала редактируемой, прежние пары теряют силу.
- FR-6: **Повторное формирование**: каждый следующий переход `draft → ready`
  формирует пары **заново**, по составу пулов **на момент этого перехода** —
  независимо от того, какие пары существовали при предыдущей фиксации. Связь
  со старым (удалённым) набором боёв не сохраняется, история прошлых наборов
  не хранится.
- FR-7: **Детерминированность**: при одном и том же составе бойцов пула
  формирование даёт один и тот же набор пар **и один и тот же их порядок**
  (FR-3a/FR-3b) — вне зависимости от того, сколько раз оно повторялось.
- FR-8: **Права и видимость**: бои формируются и видны только `ROLE_ADMIN`
  (как раскладка пулов, 0009 FR-13). В этом инкременте публичного чтения боёв
  нет — оно осознанно отложено (см. «Вне скоупа»), а не отвергнуто.

Нефункциональные:

- NFR-1: Формирование боёв — синхронная операция в рамках перевода раскладки
  в `ready` (без фоновой очереди); для типичных размеров пула (единицы–десятки
  бойцов) — мгновенно.
- NFR-2: Модель бойца-боя расширяема под будущие атрибуты (статус жизненного
  цикла, площадка, время, результат, победитель) без разрушения уже
  сформированных пар — эти атрибуты добавят следующие инкременты.

## Критерии приёмки

- **AC-1** (формирование при переходе в ready)
  - Given: раскладка номинации в `draft`; пул P с бойцами A, B, C (3 бойца)
  - When: admin переводит раскладку в `ready`
  - Then: для P сформированы ровно 3 боя: A–B, A–C, B–C; дублей и боя бойца
    самого с собой нет
- **AC-2** (пул с одним бойцом)
  - Given: раскладка в `draft`; пул P с единственным бойцом A
  - When: admin переводит раскладку в `ready`
  - Then: переход проходит успешно; для P сформировано 0 боёв; ошибки нет
- **AC-3** (пустой пул)
  - Given: раскладка в `draft`; пул P без бойцов
  - When: admin переводит раскладку в `ready`
  - Then: переход проходит успешно; для P сформировано 0 боёв; ошибки нет
- **AC-4** (изоляция пулов)
  - Given: раскладка в `draft`; пул P с бойцами A, B; пул Q с бойцами C, D
  - When: admin переводит раскладку в `ready`
  - Then: сформированы бои A–B (в P) и C–D (в Q); боёв между бойцами разных
    пулов (A–C, A–D, B–C, B–D) нет
- **AC-5** (возврат в draft удаляет бои)
  - Given: раскладка в `ready`, для пулов сформированы бои
  - When: admin возвращает раскладку в `draft`
  - Then: все бои всех пулов этой номинации удалены; экран пула их больше не
    показывает
- **AC-6** (повторное формирование по новому составу)
  - Given: раскладка была в `ready` с боями, admin вернул в `draft` и изменил
    состав пула P (добавил бойца D через DnD, 0009)
  - When: admin снова переводит раскладку в `ready`
  - Then: для P сформирован новый набор пар, учитывающий D; старые пары (без
    D) не сохранились
- **AC-7** (детерминированность)
  - Given: пул с фиксированным составом бойцов
  - When: раскладку дважды подряд переводят `ready → draft → ready` без
    изменения состава
  - Then: и набор пар, и их порядок в обоих случаях идентичны
- **AC-8** (права)
  - Given: обычный пользователь или гость
  - When: пытается прочитать или инициировать формирование боёв
  - Then: операция отклонена (`Unauthenticated`/`PermissionDenied`);
    публичного чтения боёв в этом инкременте нет
- **AC-9** (порядок — туровая структура, инвариант для любого размера пула)
  - Given: пул с N бойцами (N ≥ 2)
  - When: раскладка переведена в `ready`
  - Then: сформированный порядок боёв разбит на туры; **внутри одного тура
    ни один боец не участвует более одного раза** — это гарантировано всегда,
    независимо от N (жёсткий инвариант, не эвристика)
- **AC-10** (порядок — маленький пул, смежность на стыке туров неизбежна)
  - Given: пул с 3 бойцами (A, B, C) **либо** с 4 бойцами (A, B, C, D)
  - When: раскладка переведена в `ready`
  - Then: хотя бы на одном стыке между соседними турами оба боя делят общего
    бойца — при 3 и 4 бойцах это **математически неизбежно при любом
    порядке** (доказано: единственный бой, не пересекающийся по бойцам с
    данным, — это его же напарник по туру; переставить туры/бои так, чтобы
    убрать все стыки, невозможно). Эвристика минимизации здесь не улучшает
    результат — и не должна: это честный пол для маленьких пулов, а не её
    недоработка. Порядок при этом остаётся детерминированным (см. AC-7)
- **AC-11** (порядок — эвристика убирает смежность там, где это возможно)
  - Given: пул с 6 бойцами (n ≥ 6, чётное число)
  - When: раскладка переведена в `ready`
  - Then: итоговый порядок боёв **не содержит ни одного стыка**, где оба
    соседних боя делят общего бойца — для такого размера пула это всегда
    достижимо (между любыми двумя соседними турами существует хотя бы одна
    пара бойцов, не пересекающаяся ни с одним из бойцов последнего боя
    предыдущего тура — эвристика обязана её найти и поставить первой)

## Вне скоупа

- **Жизненный цикл отдельного боя** — старт, результат, счёт, победитель,
  статус боя (`не начат`/`идёт`/`завершён` и т.п.) — будущий инкремент.
- **Привязка боя/пула к конкретной площадке (арене) и ко времени** —
  следующий инкремент («постановка пула на арену», использует домен площадок
  из 0008). Сам порядок боёв внутри пула (FR-3a/FR-3b) формируется уже в этом
  инкременте — арена лишь исполняет готовый порядок, не пересчитывает его.
- **Посев/рейтинг при формировании пар** — round-robin не зависит от силы или
  рейтинга бойца; «удобных» первых встреч не подбираем.
- **Ручное редактирование сформированных пар** (замена соперника, добавление
  или удаление отдельного боя вручную) — не делаем; единственный способ
  получить новый набор — вернуть раскладку в `draft` и снова перевести в
  `ready`.
- **Реакция на выбытие/снятие бойца после формирования боёв** (раскладка уже в
  `ready`, у бойца уже есть назначенные бои) — технический walkover и подобная
  логика вне скоупа; полноценно решается вместе с жизненным циклом боя.
- **История/архив предыдущих наборов боёв** — при возврате в `draft` старый
  набор безвозвратно удаляется (FR-5), ничего не архивируется.
- **Публичное отображение боёв бойцам/зрителям** (на экране номинации, там
  же, где сейчас публичный состав пулов) — намеренно отложено, а не
  отвергнуто: цель — чтобы боец оперативно знал, когда и с кем его бой. Набор
  пар и их порядок (FR-3, FR-3a) в этом инкременте уже есть, но без привязки к
  площадке/времени и без статуса ЖЦ боя (оба — будущие инкременты) публичный
  экран пока не даёт бойцу ответ «когда именно» — поэтому сам публичный экран
  реализуем отдельным будущим инкрементом, когда эти данные появятся.

## Принятые решения

Зафиксировано со стейкхолдером (2026-07-15):

1. **Схема формирования — round-robin, «каждый с каждым»** внутри пула:
   каждая уникальная пара бойцов пула встречается ровно один раз. Стандартная
   схема группового этапа в HEMA/фехтовании, соответствует самому термину
   «пул».
2. **Триггер — существующий переход статуса раскладки** `draft → ready`
   (0009, FR-9), не новое отдельное действие. Симметрично: `ready → draft`
   удаляет бои.
3. **Регенерация «с нуля»**: возврат в `draft` полностью стирает бои; следующий
   `ready` формирует новый набор по актуальному на тот момент составу.
   История прежних наборов не нужна, т.к. в этом инкременте у боя ещё нет
   результата/состояния, которое было бы жаль терять.
4. **Пул с < 2 бойцами не считается ошибкой** — согласуется с тем, что сама
   раскладка (0009, AC-14) может переходить в `ready` и будучи пустой.
5. **Порядок боёв — часть этого инкремента, не «постановки на арену»**
   (уточнение стейкхолдера 2026-07-15): на арену пул ставится **целиком**, а
   не бой за боем, значит порядок исполнения — свойство самого сформированного
   набора боёв пула, а не будущей привязки к арене. Арена лишь исполняет
   готовый порядок. Механизм — туровая схема round-robin (FR-3a: жёсткий
   инвариант «не дважды в одном туре») + эвристика минимизации стыков между
   турами (FR-3b). При проработке плана выяснилось, что для маленьких пулов
   (3–4 бойца) смежность на стыке туров математически неизбежна **при любом**
   порядке — это не недоработка эвристики; а для пулов от 6 бойцов эвристика,
   наоборот, всегда может **полностью** убрать такую смежность, не просто
   уменьшить (см. AC-9/AC-10/AC-11).
6. **Бой — отдельная доменная сущность** (не просто пара id внутри пула):
   у него будет собственный жизненный цикл в будущих инкрементах. Конкретная
   архитектурная граница (новый модуль/расширение существующего, направление
   межмодульной зависимости) — решается в `plan.md`, для домена важен только
   факт: бой ссылается на пул и на двух бойцов, пул/бои admin-only, как и
   раскладка (0009 FR-13).
