# Spec: Флоу подачи заявки бойца (event-sourced)

> Артефакт SDD (ADR 0008). Описывает **ЧТО и ЗАЧЕМ**. Без технологий, API и
> схем БД — только предметная область. Реализация — в `plan.md`.

- Статус: done
- Дата: 2026-07-10

## Проблема

Турнир состоит из номинаций (дисциплин, спека 0003), но **попасть** бойцу в
номинацию сейчас нечем: нет ни заявок, ни регистрации участников. Нужен полный
путь от «боец захотел участвовать» до «боец числится в номинации турнира».

Путь неатомарен и растянут во времени: между «подал заявку» и «зарегистрирован»
проходят оплата, её подтверждение и действие секретаря. Заявка живёт как
**процесс** с историей переходов, а не как одна запись, которую перезаписывают.
Организатору важно видеть **как** заявка пришла в текущее состояние (кто и когда
подал, оплатил, подтвердил, отозвал) — для разбора спорных ситуаций и денег.

Поэтому заявка моделируется **событийно (EDD / event sourcing)**: источник
истины — неизменяемый журнал доменных событий; сама «заявка» как объект нигде
не хранится строкой, а **собирается (реконструируется) как агрегат** из своего
потока событий. Это стейкхолдерское ограничение (см. «Принятые решения», п. 1) и
первый событийный bounded context в проекте.

## Акторы

- **боец / заявитель (applicant)** — аутентифицированный пользователь (роль
  `user`, ADR 0007). Подаёт заявку в номинацию, отмечает оплату, отзывает свою
  заявку, видит статус и историю **своих** заявок. Заявка привязана к его
  `user_id`.
- **секретарь** — в MVP это **admin** (отдельной роли `ROLE_SECRETARY` не
  вводим). Подтверждает оплату и регистрирует бойца на турнире (терминальный шаг
  флоу). Видит **сводный экран** всех заявок турнира с лёгкой фильтрацией по
  статусу и номинации, а также заявки в разрезе одной номинации, с состоянием и
  историей.
- **гость** — к флоу заявок доступа не имеет: подача требует аутентификации.
  Но **читает** публичный стартовый лист номинации: имена заявленных и
  подтверждённых бойцов и счётчик «заявлено · подтверждено / лимит».

## Домен: заявка как поток событий

Заявка — это агрегат одного заявителя на **одну** номинацию (одна заявка = одна
номинация; участие в нескольких номинациях = несколько независимых заявок).

### Доменные события (факты, в прошедшем времени)

Журнал заявки состоит из событий. Каждое событие — свершившийся факт, менять
его нельзя; состояние меняется только добавлением нового события. Каждое событие
несёт инициатора (`actor_id`) и время.

| Событие                | Кто инициирует | Смысл                                            |
| ---------------------- | -------------- | ------------------------------------------------ |
| `ApplicationSubmitted` | боец           | Заявка подана в номинацию; попадает в её список  |
| `PaymentDeclared`      | боец           | Боец заявил, что оплатил (оплата ждёт проверки)  |
| `PaymentConfirmed`     | секретарь      | Оплата подтверждена                              |
| `FighterRegistered`    | секретарь      | Боец зарегистрирован (терминально, успех)        |
| `ApplicationWithdrawn` | боец           | Заявка отозвана бойцом (терминально)             |

Оплата в MVP — **чисто статусный переход**: `PaymentDeclared`/`PaymentConfirmed`
не несут суммы, способа или пруфа (платёжный провайдер и сверка сумм — вне
скоупа). Терминальный шаг `FighterRegistered` фиксирует **факт** регистрации;
фактическое создание сущности бойца в домене бойцов — кроссдоменный эффект,
который в этой фиче только обозначен как факт, а сам домен бойцов вводится
отдельной будущей фичей (см. «Принятые решения», пп. 5–6).

### Жизненный цикл (агрегат, реконструированный из событий)

Состояние — это свёртка (fold) потока событий, а не хранимое поле:

```
        (нет заявки)
             │  ApplicationSubmitted
             ▼
        ┌─────────┐  PaymentDeclared   ┌──────────────────────┐
        │ Подана  │ ─────────────────▶ │ Ожидает подтверждения│
        │(в списке│                    │       оплаты         │
        │ номин.) │                    └──────────┬───────────┘
        └────┬────┘                               │ PaymentConfirmed
             │                                     ▼
             │                          ┌────────────────────┐
             │                          │      Оплачена       │
             │                          │ (готова к регистр.) │
             │                          └──────────┬─────────┘
             │                                      │ FighterRegistered
             │                                      ▼
             │                          ┌────────────────────┐
             │                          │  Зарегистрирована   │  ← терминально
             │                          │  (боец создан)      │     (успех)
             │                          └────────────────────┘
             │
   ApplicationWithdrawn (из ЛЮБОГО нетерминального состояния,
             │            включая «Оплачена»)
             ▼
        ┌──────────┐
        │ Отозвана │  ← терминально (заявителем)
        └──────────┘
```

- Терминальные состояния: **Зарегистрирована** и **Отозвана**. Из них переходов
  нет.
- «Тут существует заявка» (по картинке) = все нетерминальные состояния: от
  `Подана` до `Оплачена`.
- Отозвать можно из **любого нетерминального** состояния, **включая
  «Оплачена»**. Возврат денег (refund) при этом — вне скоупа: финансовый вопрос
  разбирается организатором вручную. Заявка просто становится «Отозвана».

## Пользовательские сценарии

1. Как **боец**, я хочу подать заявку в номинацию турнира, чтобы заявить участие;
   после подачи моя заявка видна в списке этой номинации.
2. Как **боец**, я хочу отметить, что оплатил заявку, чтобы секретарь мог
   проверить и подтвердить оплату.
3. Как **боец**, я хочу отозвать свою заявку, пока не зарегистрирован, чтобы
   отказаться от участия.
4. Как **боец**, я хочу видеть статус своей заявки (подана / ждёт подтверждения /
   оплачена / зарегистрирован / отозвана), чтобы понимать, что происходит.
5. Как **секретарь**, я хочу видеть заявки номинации с их состоянием, чтобы
   вести приём участников.
6. Как **секретарь**, я хочу подтвердить оплату заявки, чтобы допустить её к
   регистрации.
7. Как **секретарь**, я хочу зарегистрировать оплаченного бойца, чтобы он
   числился в номинации турнира; при переполнении номинации я вижу
   предупреждение, но могу зарегистрировать.
8. Как **admin/секретарь**, я хочу видеть историю заявки (кто и когда подал,
   оплатил, подтвердил, отозвал), чтобы разбирать спорные ситуации.
9. Как **admin/секретарь**, я хочу единый экран **всех поданных заявок** турнира
   с лёгкой фильтрацией по статусу и по номинации, чтобы быстро находить и
   обрабатывать нужные заявки, не переключаясь между номинациями.
10. Как **гость/боец**, я хочу на странице номинации видеть имена **заявленных**
    и **подтверждённых** бойцов и счётчик «заявлено · подтверждено / лимит»,
    чтобы понимать состав и заполненность номинации.
11. Как **admin**, я хочу расширять лимит (soft cap) номинации, чтобы принять
    больше участников; лимит информационный и правится в управлении номинациями.

## Производные представления (счётчики и списки)

Для списков и счётчиков номинации из состояний заявок выводятся две группы
(отозванные не входят никуда):

- **Заявленные** — все **неотозванные** заявки номинации: состояния «Подана»,
  «Ожидает подтверждения оплаты», «Оплачена», «Зарегистрирована». Это общее
  число претендентов/участников.
- **Подтверждённые** — заявки с **подтверждённой оплатой и выше**: состояния
  «Оплачена» и «Зарегистрирована». Подмножество заявленных.

Счётчик номинации: «заявлено N · подтверждено M / лимит L», где `L` =
`fighter_capacity` (soft cap, спека 0003). Если `L` не задан — часть «/ лимит»
не показывается.

## Требования

Функциональные:

- FR-1: Аутентифицированный боец может подать заявку в существующую номинацию
  активного турнира. Подача фиксируется событием `ApplicationSubmitted`; заявка
  появляется в списке этой номинации в состоянии «Подана».
- FR-2: Заявка **всегда** привязана к одной номинации (а через неё — к турниру)
  и к `user_id` заявителя.
- FR-3: Боец может отметить оплату своей поданной заявки (`PaymentDeclared`) —
  заявка переходит в «Ожидает подтверждения оплаты». Оплата — статусный переход
  без денежных деталей.
- FR-4: Секретарь (admin) может подтвердить оплату (`PaymentConfirmed`) — заявка
  переходит в «Оплачена».
- FR-5: Секретарь (admin) может зарегистрировать бойца по оплаченной заявке
  (`FighterRegistered`) — заявка переходит в терминальное «Зарегистрирована».
  Фиксируется факт регистрации (создание сущности бойца — будущая фича, п. 5).
- FR-6: Боец может отозвать свою заявку (`ApplicationWithdrawn`) из **любого
  нетерминального** состояния, включая «Оплачена» — переход в терминальное
  «Отозвана». Возврат денег вне скоупа.
- FR-7: Переходы состояния определяются **только** допустимыми событиями из
  текущего состояния. Недопустимый переход (напр. подтвердить оплату у заявки,
  где оплата не заявлена; любое действие над терминальной заявкой) отклоняется,
  состояние не меняется.
- FR-8: Заявитель видит статус и историю **своих** заявок (`ListMyApplications`).
  Секретарь/admin видят все заявки турнира, сгруппированные по номинациям, с
  состоянием и историей.
- FR-9: Источник истины — журнал событий; текущее состояние заявки и любые
  списки/статусы **выводятся** из событий (реконструкция агрегата), а не
  хранятся как отдельная перезаписываемая запись (EDD, п. 1).
- FR-10: История заявки неизменяема: события не редактируются и не удаляются;
  «отмена» действия — это новое компенсирующее событие, а не правка прошлого.
- FR-11: Права: подать/оплатить/отозвать может только сам заявитель своей
  заявки; подтвердить оплату и зарегистрировать — только admin (секретарь).
- FR-12: Запрет активного дубля: у одного `user_id` не может быть двух
  **активных** (нетерминальных) заявок в одну номинацию. После отзыва можно
  подать заявку в эту номинацию заново; после регистрации — нельзя (уже
  участник).
- FR-13: При регистрации, если число уже зарегистрированных в номинации
  достигло/превысило `fighter_capacity` (спека 0003), секретарю возвращается
  **мягкое предупреждение** о переполнении, но регистрация не блокируется.
  Если `fighter_capacity` не задан — предупреждения нет.
- FR-14: **Сводный экран заявок турнира** (admin): список всех заявок турнира с
  **лёгкой фильтрацией** по статусу (состояние заявки) и по номинации. Фильтры
  опциональны и комбинируются; без фильтров — все заявки турнира. По каждой
  заявке видны номинация, имя заявителя и состояние.
- FR-15: На странице номинации показываются **имена бойцов**: список заявленных
  и список подтверждённых (см. «Производные представления»). Имя = отображаемое
  имя (`display_name`) пользователя-заявителя; резолвится из домена `auth` (не
  дублируется в заявке). Отозванные заявки в списки не входят.
- FR-16: Рядом с номинацией показывается **счётчик** «заявлено N · подтверждено
  M / лимит L». `N`/`M` выводятся из состояний заявок (проекция), `L` =
  `fighter_capacity`; при незаданном лимите «/ лимит» скрывается. Счётчики и
  списки имён **видны публично** (стартовый лист номинации).
- FR-17: Лимит номинации (soft cap) **расширяем**: admin меняет
  `fighter_capacity` через управление номинациями (спека 0003). Лимит
  информационный; превышение не блокирует регистрацию (FR-13).

Нефункциональные:

- NFR-1: Чтение списков заявок номинации и статуса «моих заявок» должно быть
  дешёвым для UI (не заставлять клиента сворачивать сырые события) — состояние
  предоставляется уже реконструированным.
- NFR-2: Модель расширяема: добавление нового типа события/перехода в будущем
  (напр. отклонение заявки секретарём, возврат оплаты, роль `ROLE_SECRETARY`,
  создание сущности бойца) не должно ломать уже записанную историю.
- NFR-3: Одновременные действия над одной заявкой не должны приводить к
  недопустимому состоянию (напр. двойная регистрация, регистрация после отзыва,
  два активных дубля).

## Критерии приёмки

- **AC-1** (подача → список номинации)
  - Given: есть активный турнир с номинацией, боец аутентифицирован
  - When: боец подаёт заявку в номинацию
  - Then: записан факт `ApplicationSubmitted`; заявка видна в списке номинации в
    состоянии «Подана» и в «моих заявках» бойца
- **AC-2** (оплата бойцом)
  - Given: заявка бойца в состоянии «Подана»
  - When: боец отмечает оплату
  - Then: заявка переходит в «Ожидает подтверждения оплаты»; в истории — факт
    `PaymentDeclared`
- **AC-3** (подтверждение секретарём)
  - Given: заявка «Ожидает подтверждения оплаты»
  - When: admin подтверждает оплату
  - Then: заявка переходит в «Оплачена»
- **AC-4** (регистрация)
  - Given: заявка «Оплачена»
  - When: admin регистрирует бойца
  - Then: заявка становится «Зарегистрирована» (терминально); зафиксирован факт
    `FighterRegistered`
- **AC-5** (отзыв из «Оплачена»)
  - Given: заявка бойца в состоянии «Оплачена»
  - When: боец отзывает заявку
  - Then: заявка становится «Отозвана» (терминально) и не может быть
    зарегистрирована
- **AC-6** (недопустимый переход)
  - Given: заявка «Подана» (оплата не заявлена)
  - When: admin пытается подтвердить оплату
  - Then: операция отклоняется (`FailedPrecondition`); состояние не меняется
- **AC-7** (действие над терминальной заявкой)
  - Given: заявка «Отозвана» (или «Зарегистрирована»)
  - When: кто-либо пытается оплатить/подтвердить/зарегистрировать/отозвать
  - Then: операция отклоняется
- **AC-8** (права заявителя)
  - Given: заявитель A подал заявку
  - When: другой боец B пытается оплатить/отозвать заявку A
  - Then: операция отклоняется (`PermissionDenied`)
- **AC-9** (права секретаря)
  - Given: обычный боец (роль `user`)
  - When: он пытается подтвердить оплату или зарегистрировать бойца
  - Then: операция отклоняется (`PermissionDenied`)
- **AC-10** (история/аудит)
  - Given: заявка прошла путь подача → оплата → подтверждение → регистрация
  - When: admin открывает историю заявки
  - Then: он видит все события по порядку с временем и инициатором
- **AC-11** (реконструкция состояния)
  - Given: у заявки в журнале несколько событий
  - When: запрашивается текущее состояние заявки
  - Then: оно вычислено из полного потока событий и соответствует последнему
    допустимому переходу
- **AC-12** (запрет активного дубля)
  - Given: у бойца уже есть активная заявка в номинацию
  - When: он подаёт вторую заявку в ту же номинацию
  - Then: операция отклоняется (`AlreadyExists`)
- **AC-13** (повторная заявка после отзыва)
  - Given: боец отозвал заявку в номинацию (терминальная «Отозвана»)
  - When: он подаёт новую заявку в эту же номинацию
  - Then: заявка принимается (активного дубля нет)
- **AC-14** (мягкое предупреждение о переполнении)
  - Given: в номинации `fighter_capacity = N` и уже N зарегистрированных
  - When: admin регистрирует ещё одного бойца
  - Then: регистрация проходит; в ответе — признак/предупреждение о переполнении
- **AC-15** (сводный экран с фильтрами)
  - Given: у турнира заявки в разных номинациях и разных состояниях
  - When: admin открывает сводный экран заявок и фильтрует по статусу «Оплачена»
    и/или по конкретной номинации
  - Then: список содержит только подходящие заявки (по имени заявителя,
    номинации, состоянию); без фильтров — все заявки турнира
- **AC-16** (имена и счётчик на странице номинации)
  - Given: у номинации `fighter_capacity = 16`, 12 неотозванных заявок, из них 8
    с подтверждённой оплатой и выше
  - When: гость открывает страницу номинации
  - Then: он видит счётчик «заявлено 12 · подтверждено 8 / 16», список имён
    заявленных (12) и подтверждённых (8)
- **AC-17** (отозванная не в списках/счётчиках)
  - Given: боец отозвал заявку в номинацию
  - When: гость открывает страницу номинации
  - Then: его имя не показано ни в заявленных, ни в подтверждённых; счётчики его
    не считают
- **AC-18** (лимит не задан)
  - Given: у номинации `fighter_capacity` не задан
  - When: гость открывает страницу номинации
  - Then: показан счётчик «заявлено N · подтверждено M» без части «/ лимит»

## Вне скоупа

- **Интеграция платёжного провайдера** (эквайринг, вебхуки, суммы, чеки).
  Оплата — статусный переход (боец заявил → секретарь подтвердил).
- **Возвраты (refund)** и связанная финансовая логика (в т.ч. при отзыве
  оплаченной заявки — разбирается вручную).
- **Отклонение/бан заявки секретарём** (событие-отказ вместо подтверждения) —
  будущее расширение (NFR-2).
- **Жёсткий лимит ёмкости номинации** — в MVP только мягкое предупреждение
  (FR-13); блокирующая проверка и разрешение гонок — будущая фича.
- **Полноценный домен бойцов** (сущность бойца, профиль, клуб, рейтинг,
  редактирование, реальное создание при регистрации) — здесь только терминальный
  **факт** `FighterRegistered`; сам домен и кроссдоменный вызов — отдельная
  будущая фича.
- **Роль `ROLE_SECRETARY`** — в MVP секретарь = admin; выделение роли — будущий
  инкремент (модель это допускает, NFR-2).
- **Снапшоты/проекции как оптимизация** event store — детали реализации, уровень
  `plan.md`, не домен.
- **Мультитурнирность** — как и в 0003, модель не мешает, но выбор турнира вне
  скоупа.

## Принятые решения

Зафиксировано со стейкхолдером (2026-07-10):

1. **EDD / event sourcing для заявки.** Заявка не хранится строкой; источник
   истины — неизменяемый журнал доменных событий, из которого заявка
   реконструируется как агрегат в репозитории. Вводит **первый событийный
   bounded context** и требует нового ADR по Event-Driven Design (плейсхолдер
   «События» в шаблоне `plan.md` для этой фичи наполняется реально).
2. **Заявитель — аутентифицированный пользователь** (роль `user`). Заявка
   привязана к `user_id`; «мои заявки» и права «своей заявки» — по нему.
3. **Секретарь = admin** в MVP. Отдельную роль `ROLE_SECRETARY` не вводим (вне
   скоупа).
4. **Одна заявка = одна номинация.** Несколько номинаций — несколько независимых
   заявок. Запрет активного дубля на пару (`user_id`, `nomination_id`).
5. **Терминал — только факт `FighterRegistered`.** Полноценная сущность бойца и
   кроссдоменное создание — будущая фича; здесь флоу лишь доводит заявку до
   «Зарегистрирована» и фиксирует факт.
6. **Создание бойца — кроссдоменный эффект.** Когда домен бойцов появится,
   регистрация будет инициировать создание бойца в номинации через API того
   домена (модульные границы, ADR 0002) — не прямой записью в чужую схему. Сейчас
   — только факт.
7. **Оплата — статусный переход** без суммы/способа/пруфа (провайдер вне скоупа).
8. **Отзыв — из любого нетерминального состояния**, включая «Оплачена»; refund
   вне скоупа.
9. **Ёмкость номинации — мягкое предупреждение** при регистрации, не
   блокирующее.
10. **«Подтверждённые» = оплата подтверждена и выше** (состояния «Оплачена» +
    «Зарегистрирована»); **«заявленные» = все неотозванные**. Отозванные не
    входят в счётчики/списки. Счётчик — обе цифры: «заявлено · подтверждено /
    лимит».
11. **Имена бойцов = `display_name`** пользователя-заявителя, резолвятся из
    домена `auth` (межмодульная граница, ADR 0002) — в заявке не дублируются.
12. **Имена и счётчики видны публично** (стартовый лист номинации) — осознанно
    принятая приватность: заявитель, подавая заявку, попадает в публичный список
    участников номинации.
13. **Сводный экран заявок турнира для admin** — с лёгкой фильтрацией по статусу
    и номинации (в дополнение к списку по одной номинации).
