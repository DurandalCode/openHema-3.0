# Spec: Статусная модель номинации

> Артефакт SDD (ADR 0008). Описывает **ЧТО и ЗАЧЕМ**. Без технологий, API и
> схем БД — только предметная область. Реализация — в `plan.md`.

- Статус: done
- Дата: 2026-07-18

## Проблема

Сегодня у **номинации** нет собственной статусной модели. Номинация
(спека 0003) — статический справочник дисциплин турнира с CRUD + порядок; её
состояние в жизненном цикле турнира никак не отражено. Между тем у номинации
есть очевидные фазы:

- **приём заявок** — бойцы подают заявки (спека 0005), секретарь их
  подтверждает и регистрирует;
- **приём заявок завершён** — admin закрывает приём вручную, новых заявок
  больше не принимается;
- **раскладка по пулам** — admin начал раскладку бойцов по пулам (спека 0009),
  заявки в номинацию тоже не принимаются по существу;
- далее — собственно бои (`active`/`finished`), но это будущие инкременты
  (0010/0011 и дальше).

Сейчас граница «приём заявок открыт/закрыт» нигде не зафиксирована и не
защищена: боец может подать заявку в номинацию, по которой уже идёт раскладка
(или которую admin формально закрыл), – это бессмысленное состояние, которое
придётся разбирать руками. А гость и боец не видят на публичной странице
номинации, что приём уже завершён.

Нужно ввести у номинации **статус жизненного цикла** (два публичных значения —
«приём заявок открыт» / «приём заявок закрыт»), который:

1. даёт admin'у возможность **вручную закрыть приём заявок** номинации;
2. автоматически переводит номинацию в «закрытый» статус, как только
   **раскладка по пулам началась** — конкретный триггер: **первый посад
   бойца в пул** (`assign`, спека 0009);
3. даёт admin'у **открыть приём заявок обратно** — но только если закрытие
   произошло вручную (не автоматически от раскладки); автоматическое закрытие
   от раскладки откатывается автоматически, как только раскладка полностью
   удалена (см. «Принятые решения» №4 и FR-4);
4. делает статус видимым: публично на странице номинации и в админке, и
   используется как **гейт при подаче заявки** (номинация закрыта →
   `SubmitApplication` отклоняется, спека 0005).

Триггер «раскладка началась» намеренно автоматический: переход к посадке
бойцов в пулы — это уже решение организатора «состав finalized, переходим к
боевому распределению», и держать параллельно открытый приём заявок — ошибка.

### Замечание о «сводном» статусе `CLOSED`

Различение причины закрытия (admin вручную vs раскладка началась) **не
публично**: пользователь (гость, боец) видит только `OPEN`/`CLOSED`. Однако
система должна различать их **внутренне**, чтобы корректно гейтить
`ReopenRegistration` (FR-4) и автоматически откатывать `CLOSED → OPEN` при
удалении раскладки (FR-5). Конкретный механизм (отдельное поле-флаг,
внутреннее перечисление-причина, или проверка состояния `pool` при RPC) —
вопрос `plan.md`.

## Акторы

- **admin (секретарь)** — управляет статусом номинации: вручную закрывает и
  открывает приём заявок через отдельные RPC. Перевод номинации в
  «раскладка началась» происходит автоматически от действий admin'а в модуле
  `pool` (первый `assign` бойца в пул), отдельного RPC на этот переход нет.
  Роль `ROLE_ADMIN`.
- **боец / заявитель (applicant)** — видит статус номинации публично (`OPEN`
  / `CLOSED`); подача заявки (`SubmitApplication`) возможна **только** когда
  номинация в `OPEN`; иначе — осмысленная ошибка.
- **гость** — видит статус номинации на публичной странице турнира (бейдж
  «Приём заявок» / «Приём заявок завершён»). На подачу заявки возможности нет
  в любом случае (она требует аутентификации).

## Пользовательские сценарии

1. Как **admin**, я хочу вручную закрыть приём заявок в номинацию, чтобы
   зафиксировать состав и не принимать новые заявки.
2. Как **admin**, я хочу при необходимости **открыть приём заявок обратно**
   после ручного закрытия, если закрытие было моим ручным действием (а не
   автоматическим от раскладки).
3. Как **admin**, начав раскладку бойцов по пулам и посадив первого бойца в
   пул, я ожидаю, что номинация автоматически переходит в статус `CLOSED` и
   приём заявок закрывается — без отдельного ручного шага, и без возможности
   открыть его обратно вручную, пока раскладка не удалена полностью.
4. Как **admin**, я хочу, чтобы при полном удалении раскладки (нечего
   распределать — все бойцы вынуты и пулы удалены) номинация автоматически
   вернулась в `OPEN` — это даёт мне путь «случайно открыл раскладку — откатить».
5. Как **боец**, я хочу при попытке подать заявку в закрытую номинацию
   получить осмысленную ошибку «приём заявок завершён», а не молчаливый
   «зависший» ответ.
6. Как **гость**, я хочу видеть на публичной странице номинации бейдж/подпись,
   что приём заявок завершён, чтобы не пытаться подать заявку впустую.
7. Как **admin**, я хочу видеть статус каждой номинации на экране управления
   номинациями, чтобы быстро понимать, какие номинации ещё принимают заявки,
   какие — уже нет.

## Требования

Функциональные:

- FR-1: У номинации появляется **статус жизненного цикла** — публичное
  перечисление из двух значений + закладки под бои:
  - **`OPEN`** (default) — приём заявок открыт;
  - **`CLOSED`** — приём заявок закрыт (причина не публична: ручное закрытие
    admin'ом либо автозакрытие от старта раскладки);
  - **`ACTIVE` / `FINISHED`** — enum-закладки под будущую фазу боёв. В этом
    инкременте **не реализуются**: переходов в коде нет, поведения нет. Как
    и `active`/`finished` в 0009/0010, оставлены ради целостности контракта
    и расширяемости без ломки данных.
- FR-2: При создании номинации (спека 0003) её статус — `OPEN`.
- FR-3: admin может **вручную закрыть и открыть приём заявок** через
  отдельные RPC `CloseRegistration` и `ReopenRegistration` (пара). Один RPC =
  один переход, без положения статуса в общий `UpdateNomination`.
- FR-4: **`ReopenRegistration`** гейт — разрешён, только если **оба** условия
  верны:
  1. `CLOSED` установлен вручную (закрытие сделал admin через
     `CloseRegistration`, а не автоматически от раскладки, см. FR-5);
  2. на номинации **сейчас нет ни одного распределённого бойца** (раскладка
     не идёт).
  - Отклоняется `FailedPrecondition`, если хотя бы одно условие не
    выполняется — в т.ч. пограничный случай: admin вручную закрыл приём (нет
    раскладки), но **затем всё же начал раскладку** (модуль `pool` не
    проверяет статус номинации при `assign` — гейт FR-7 есть только у
    `SubmitApplication`); раз распределённые бойцы есть — `Reopen`
    заблокирован, даже хотя причина закрытия осталась «вручную». Как только
    раскладка полностью убрана (0 распределённых) — `Reopen` снова доступен,
    причина по-прежнему «вручную» (не апгрейдится в «от раскладки», FR-5 в
    этом случае — no-op, см. FR-10).
  - Конкретный механизм «как мы знаем, что `CLOSED` — от раскладки, и сколько
    сейчас распределённых бойцов» — `plan.md` (внутренний флаг/причина плюс
    проверка текущего состояния `pool`).
- FR-5: **Автоматический переход `OPEN → CLOSED`** срабатывает, когда в
  модуле `pool` (спека 0009) появляется **первый распределённый боец
  номинации** — до мутации распределённых бойцов не было (0), после —
  стало ≥1. Конкретные правила срабатывания:
  - до первого распределения — статус номинации не меняется (даже если
    созданы пустые пулы, даже если открыт экран раскладки — lazy-init строки
    `pool_layout` из 0009 не является триггером);
  - переход 0 → ≥1 распределённых бойцов → номинация переходит в `CLOSED`
    автоматически (синхронным вызовом pool-сервиса к API модуля `nomination`,
    см. FR-10); срабатывает независимо от того, какая мутация пула его
    вызвала (`AssignFighter` из «нераспределённые», `AutoDistribute`, `Undo`,
    восстанавливающий пул с бойцами, и т.п. — см. FR-10);
  - идентификатор «кто закрыл» = системный/пул-сервис, не admin
    (для будущего аудита — см. «Вне скоупа»).
- FR-6: **Автоматический откат `CLOSED → OPEN`** срабатывает симметрично —
  когда число распределённых бойцов номинации падает с ≥1 до 0: все пулы
  номинации удалены и нет ни одного членства бойца в пуле. Срабатывает
  независимо от того, какая мутация пула его вызвала (`DeletePool`,
  `ResetLayout`, `UnassignFighter` последнего бойца, `Undo`, откатывающий
  `AutoDistribute`, и т.п. — см. FR-10). Аналогично — синхронный вызов
  pool-сервиса к `nomination`. Если `CLOSED` был установлен **вручную**, а
  не от раскладки — откат не срабатывает (ручное закрытие не анулируется
  cleanup'ом пустой раскладки). Защита: откаты касательно только автоматической
  причины закрытия.
- FR-7: **Подача заявки** (`SubmitApplication`, спека 0005) разрешена только
  если номинация в статусе `OPEN`. Иначе — `FailedPrecondition` с
  осмысленным сообщением. Это кросс-модульное изменение: модуль `application`
  уже обращается к `NominationProvider` (0005, порт поверх
  `nomination`-сервиса в `internal/platform`) — расширяем контракт тем, что
  `NominationProvider` отдаёт статус, а сервис `application` проверяет гейт
  наряду с проверкой существования номинации.
  - Закрытие приёма влияет **только на новую подачу** `SubmitApplication`.
    Уже поданные (нетерминальные) заявки проходят обычный флоу 0005 без
    изменений: боец может отозвать заявку (`ApplicationWithdrawn`),
    отметить оплату (`PaymentDeclared`), секретарь может подтвердить оплату
    (`PaymentConfirmed`) и зарегистрировать бойца (`FighterRegistered`).
- FR-8: Статус номинации **видим публично** — в `Nomination` сообщении proto
  появляется поле `status`; публичное чтение (0003, `ListNominations`,
  `GetNomination`) отдаёт статус без auth.
- FR-9: Статус номинации **видим в админке** — на экране управления
  номинациями и/или в карточке номинации отображается текущий статус, а
  также UI-кнопки «Закрыть приём»/«Открыть приём». Доступность кнопок:
  - при `OPEN` — активна «Закрыть приём», «Открыть приём» — недоступна;
  - при `CLOSED` (внутренне известно, что ручное закрытие) **и** сейчас нет
    распределённых бойцов — активна «Открыть приём», «Закрыть приём» —
    недоступна;
  - при `CLOSED` (внутренне известно, что от раскладки, **или** ручное
    закрытие, но раскладка сейчас всё же активна — см. FR-4) — обе
    недоступны (возвращать приём заявок можно только полным удалением
    раскладки в модуле пулов);
  - при `ACTIVE`/`FINISHED` (будущее) — ручные переходы приёма не имеют
    смысла, обе недоступны.
- FR-10: **Связь `pool → nomination`** реализуется **синхронным вызовом
  pool-сервиса к API модуля `nomination`** — триггер **не привязан к
  конкретному имени RPC**, а к результирующему состоянию: после любой
  pool-мутирующей операции, способной изменить число распределённых бойцов
  номинации, pool-сервис проверяет, не пересекло ли это число границу 0 ↔ ≥1
  (FR-5/FR-6), и синхронно уведомляет `nomination`, если да. Это касается
  **всех** мутирующих RPC `PoolAdminService`, а не только «очевидных»:
  `AssignFighter`, `UnassignFighter`, `DeletePool`, `ResetLayout`,
  `AutoDistribute`, `Undo` (последний особенно важен — он может одним
  действием восстановить/удалить целый пул со всеми бойцами, см. 0009 FR-7a).
  Без прямого кросс-схемного доступа к PG-схеме `nomination`
  (ADR 0002); межмодульное взаимодействие через публичный
  admin-сервис `nomination`. EDD (событийная шина) для этого триггера **не
  вводится** (по предварительной оценке — overhead; модуль `pool` — не EDD
  по решению 0017 из 0009, `nomination` — не EDD по своей природе). Если
  позже появится межмодульная событийная шина — переход на неё отдельным ADR.
- FR-11: **Связь с существующей статусной моделью пулов** (спека 0009/0011).
  У номинации есть собственный статус `OPEN`/`CLOSED` (этот инкремент); у
  раскладки пула — `draft`/`ready` (0009, 0011 урезал до двух), у отдельного
  пула — `NOT_READY`/`READY`/`PREPARING` (0011). Статус номинации — это
  надстройка над фактом «раскладка началась» (есть распределённые бойцы), а
  не замена статусов раскладки; они ортогональны (номинация vs её раскладка).
- FR-12: Управление статусом (закрыть/открыть приём) — только `ROLE_ADMIN`.
  Перевод в `CLOSED` от раскладки `pool`-сервисом — внутренний системный
  вызов; обычный пользователь / гость исполнить не может (выполняется на
  серверной стороне как часть RPC `pool`, который сам уже требует admin).

Нефункциональные:

- NFR-1: Чтение статуса номинации — дешёвое, вписывается в уже существующие
  публичные запросы (список и одна номинация), не порождает лишних сетевых
  обращений на публичной странице (статус хранится в поле номинации —
  выбирается вместе с ней).
- NFR-2: Гейт при подаче заявки — синхронная проверка в сервисе
  `application` (один обращение к `NominationProvider` в рамках
  `SubmitApplication`, который и так его делает в 0005). Без отдельной
  блокировки/транзакции.
- NFR-3: Модель расширяема под будущие статусы (`ACTIVE`/`FINISHED` — фаза
  боёв) без ломки контракта и данных (см. enum-закладку 0009/0010).
- NFR-4: Существующие данные **не мигрируются искусственно**: на момент
  инкремента система не в проде, нет требований к backfill. Миграция схемы
  добавляет колонку статуса с `DEFAULT OPEN`; существующие номинации,
  у которых уже есть распределённые бойцы в пулах, корректность на
  «текущий момент» получает при следующей мутации пула (`assign`/удаление) —
  без отдельной backfill-процедуры. По мере разработки это естественным
  образом консистентно.

## Критерии приёмки

- **AC-1** (default статус при создании)
  - Given: admin создаёт номинацию
  - When: номинация сохраняется
  - Then: её статус — `OPEN`
- **AC-2** (публичное чтение отдаёт статус)
  - Given: у активного турнира есть номинации со статусами `OPEN` и `CLOSED`
  - When: гость запрашивает список / одну номинацию
  - Then: в ответе есть поле `status`, отражающее жизненную фазу
- **AC-3** (ручное закрытие приёма)
  - Given: номинация в статусе `OPEN`, раскладки бойцов по пулам нет
  - When: admin вызывает `CloseRegistration`
  - Then: статус номинации — `CLOSED`
- **AC-4** (ручное открытие приёма — после ручного закрытия)
  - Given: номинация в статусе `CLOSED` (установлен ручным `CloseRegistration`,
    раскладки нет)
  - When: admin вызывает `ReopenRegistration`
  - Then: статус номинации — `OPEN`
- **AC-5** (подача заявки в открытую номинацию — разрешена)
  - Given: номинация в статусе `OPEN`
  - When: боец подаёт заявку
  - Then: заявка принимается (далее — по флоу 0005)
- **AC-6** (подача заявки в закрытую — отклонена)
  - Given: номинация в статусе `CLOSED` (любая причина)
  - When: боец подаёт заявку
  - Then: `FailedPrecondition` с осмысленным сообщением «приём заявок
    завершён»
- **AC-7** (автоматический переход при первом посаде бойца в пул)
  - Given: номинация в любом «до-раскладочного» статусе (`OPEN` или
    `CLOSED` от ручного закрытия); пулы могут быть созданы, но бойцов в пулах
    нет
  - When: admin выполняет первый `assign` бойца в пул номинации
  - Then: статус номинации становится `CLOSED` (системное уведомление от
    pool-сервиса к nomination-сервису, в той же/согласованной транзакции)
- **AC-8** (пустые пулы не триггерят)
  - Given: номинация в `OPEN`; admin создал пустые пулы (или просто открыл
    экран раскладки, создав строку `pool_layout`)
  - When: никаких `assign`'ов не было
  - Then: статус номинации остаётся `OPEN`
- **AC-9** (защита `ReopenRegistration` после автоматического закрытия)
  - Given: номинация `CLOSED` от раскладки (есть распределённые бойцы)
  - When: admin пытается вызвать `ReopenRegistration`
  - Then: `FailedPrecondition` с осмысленным сообщением (раскладка активна —
    вернуть приём можно только полным удалением раскладки)
- **AC-10** (автоматический откат при полном удалении раскладки)
  - Given: номинация `CLOSED` от раскладки; admin удаляет всех бойцов из
    пулов и сами пулы (или делает «сброс раскладки», FR-4a из 0009) — в итоге
    распределённых бойцов и пулов нет
  - When: pool-сервис детектит «раскладка полностью удалена»
  - Then: статус номинации автоматически становится `OPEN` (т.к. закрытие
    было от раскладки, автоматический откат валиден)
- **AC-11** (ручное закрытие не откатывается пустой раскладкой)
  - Given: admin вручную закрыл приём (`CLOSED`, ручной), раскладки нет
  - When: pool-сервис рапортует «раскладка пуста»
  - Then: статус номинации остаётся `CLOSED` (откат касается только
    автоматической причины закрытия)
- **AC-12** (кнопки в админке по статусу)
  - Given: admin смотрит карточку/строку номинации
  - When: статус `OPEN`
  - Then: доступна кнопка «Закрыть приём»; «Открыть приём» — недоступна
  - And: при `CLOSED` (внутренне известно, что ручное) и 0 распределённых
    бойцов — наоборот
  - And: при `CLOSED` (внутренне известно, что от раскладки, либо ручное с
    активной раскладкой — AC-16) — обе недоступны
- **AC-13** (права на управление)
  - Given: обычный пользователь или гость
  - When: пытается вызвать `CloseRegistration`/`ReopenRegistration`
  - Then: `Unauthenticated`/`PermissionDenied`
- **AC-14** (public UI бейдж)
  - Given: гость открывает публичную страницу номинации
  - When: статус `CLOSED`
  - Then: рядом с номинацией — бейдж «Приём заявок завершён» (или
    эквивалент); кнопки/ссылки «подать заявку» скрываются или показываются как
    недоступные
- **AC-15** (уже поданные заявки живут флоу 0005 при закрытой номинации)
  - Given: номинация `CLOSED`, у бояца P есть нетерминальная заявка в эту
    номинацию
  - When: P отмечает оплату (`PaymentDeclared`) / отзывает заявку
    (`ApplicationWithdrawn`); секретарь подтверждает оплату /
    регистрирует бойца
  - Then: эти операции выполняются по обычному флоу 0005 — закрытие приёма
    их не блокирует
- **AC-16** (ручное закрытие + раскладка всё же началась → Reopen заблокирован)
  - Given: admin вручную закрыл приём (`CLOSED`, ручной), раскладки ещё не
    было; затем admin всё же выполнил `assign` бойца в пул этой номинации
    (модуль `pool` не проверяет статус номинации — гейт есть только у
    `SubmitApplication`, FR-7)
  - When: admin пытается вызвать `ReopenRegistration`
  - Then: `FailedPrecondition` — раскладка сейчас активна (есть распределённые
    бойцы), несмотря на то что причина закрытия осталась «ручной» (FR-4)
  - And: когда admin полностью убирает раскладку (0 распределённых бойцов) —
    `ReopenRegistration` снова разрешён (причина по-прежнему «ручная», не
    апгрейднулась в «от раскладки» — FR-5/FR-10 no-op, пока номинация уже
    `CLOSED`)

## Вне скоупа

- **Активная фаза боёв номинации** (`ACTIVE`/`FINISHED`) — enum- закладка
  под будущий инкремент, переходы и поведение не реализуются (как 0009/0010).
- **Различение причины `CLOSED` в публичном API** — внутреннее; гость и
  боец видят только `OPEN`/`CLOSED`. Внутренняя причина — деталь реализации,
  не контракт.
- **Массовое закрытие приёма заявок по всем номинациям турнира** — операция
  «закрыть весь турнир разом» обычный admin здесь не делает (каждая
  номинация управляется отдельно).
- **Уведомления бойцов о закрытии приёма** — вне скоупа.
- **Гейт статуса на `EditApplication`** (перенос заявки в другую номинацию,
  спека 0006) — намеренно **не гейтится** статусом `CLOSED`. Это осознанное
  admin-действие (сам admin уже прошёл авторизацию и явно переносит заявку),
  а не обход гейта заявителем — в отличие от `SubmitApplication` (FR-7),
  которую заявитель может вызвать сам. Симметрично другим admin-override
  возможностям в этом инкременте.
- **История изменения статуса номинации** (кто/когда закрывал приём) — вне
  скоупа; в этом инкременте храним только текущее значение (+ внутренний
  флаг причины при необходимости). Если позже понадобится аудит —
  EDD-расширение модуля `nomination` или отдельный ADR.
- **Миграция существующих данных** — система не в проде; явный backfill
  статусов существующим номинациям не пишем. Миграция ограничивается схемой
  (колонка со `DEFAULT OPEN` + автоматический converge при следующих
  мутациях пула).
- **Связь `nomination` → `pool` для чтения статуса** — статус номинации
  хранится (а не вычисляется на чтение), поэтому отдельного read-порта в
  `pool` не появляется.
- **Уведомление nomination → application об изменении статуса** — `application`
  при `SubmitApplication` читает текущий статус номинации синхронно через
  `NominationProvider`; буфера/кеша/события нет.

## Принятые решения

Зафиксировано со стейкхолдером (2026-07-18):

1. **Состав enum** — публичное перечисление `OPEN`/`CLOSED`, плюс
   enum-закладки `ACTIVE`/`FINISHED` (без переходов). Различение «manual» vs
   «drawing» причины не публично — внутри реализации (флаг/причина), нужно
   только для гейта `ReopenRegistration` (FR-4) и условных авто-откатов (FR-6).
2. **Форма операции** — отдельные RPC `CloseRegistration` /
   `ReopenRegistration`. Поле `status` в `UpdateNomination` не добавляем:
   статусная логика отделена от профиля номинации (CLEAN, отдельный гейт по
   правам, проще тесты переходов).
3. **Триггер «раскладка началась»** — **первый посад бойца в пул** (`assign`,
  FR-5 из 0009). Создание пустых пулов и lazy-init строки `pool_layout`
   триггерами не являются (AC-8). Мотивация: бои укладываются реальными
   бойцами, а не структурой пула как таковой; до первого распределения
   admin ещё может и закрыть вручную, и спокойно продолжать приём заявок,
   раскладывая «скелет» пулов.
4. **Обратимость из `CLOSED` (drawn)** — вручную вернуть нельзя (FR-4).
   Автоматический откат `CLOSED → OPEN` срабатывает только при полном
   удалении раскладки (нет пулов и нет членств). Если `CLOSED` установлен
   вручную — откат не срабатывает (FR-6). Это создаёт внутри реализации
   потребность отличать «ручную» причину от «автоматической» — выбор
   механизма (отдельное поле-флаг, отдельное внутреннее перечисление
   причины, синхронная проверка состояния `pool` при RPC) — `plan.md`.
   **Уточнение при ревалидации (2026-07-20):** `ReopenRegistration` гейтится
   **и** причиной («ручная»), **и** текущим состоянием раскладки (0
   распределённых бойцов) — иначе admin, вручную закрывший приём, а затем
   всё же начавший раскладку (pool не проверяет статус номинации при
   `assign`, гейт есть только у `SubmitApplication`, FR-7), мог бы открыть
   приём заявок обратно, пока раскладка активна (AC-16). Причина при этом не
   апгрейдится в «от раскладки» — переход 0→≥1, застающий номинацию уже
   `CLOSED`, для FR-5/FR-10 остаётся no-op.
5. **Синхронизация между `pool` и `nomination`** — **хранимый статус** +
   синхронный вызов pool-сервиса к API модуля `nomination` при mutation
   пула, влияющей на триггеры (FR-10). EDD (событийная шина) не вводится:
   по 0017 `pool` — не EDD, по своей природе `nomination` — не EDD; ради
   одного триггера вводить события — overhead. Зависимость направлена
   `pool → nomination` (обратная к уже существующей `application →
   nomination`); прямых обращений к чужим PG-схемам нет.
6. **Влияние на уже поданные заявки** — закрытие приёма блокирует **только**
   новую `SubmitApplication`. Нетерминальные заявки продолжают обычный флоу
   0005 (отзыв, оплата, подтверждение, регистрация). Мотивация: минимальное
   вмешательство в событийный флоу `application`; регистрация
   оплаченных заявок не блокируется — иначе терминал «Оплачена» не
   дойдёт до «Зарегистрирована».
7. **Миграция существующих данных** — система не в проде, backfill не
   пишем; миграция ограничивается схемой (`DEFAULT OPEN` для новой колонки
   статуса). Существующие «раскладки с бойцами» консистентно converge'нут
   при следующей мутации `pool` (assign или удаление). На тестовых данных
   при разработке допускается ручная правка / пересоздание.
