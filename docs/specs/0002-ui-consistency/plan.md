## Обзор решения

Чисто презентационный рефактор web-слоя (`/web`), синхронный, **без** новых
модулей, RPC и без изменения контрактов. Работаем только в FSD-слоях
`shared/ui`, `shared/config`, `entities`, `features`, `widgets` и роутах
`src/app/**` (не BFF-ручки). Три направления: (1) добавить недостающие
примитивы дизайн-системы и перевести на них дублирующуюся ad-hoc-разметку;
(2) убрать пути в никуда (битые якоря, реклама несуществующих фич); (3) удалить
мёртвый код и унифицировать сквозные affordances (загрузка, «назад», пустые
состояния). Границы импортов FSD (ADR 0005) и правила тем (ADR 0005 §4)
сохраняются.

### 1. Новые примитивы дизайн-системы — `shared/ui/`

Добавляются через `pnpm dlx shadcn@latest add <c>` (коммитятся в `shared/ui`,
это наш код, не gen — ADR 0005 §3), стиль `new-york`, палитра `zinc`:

- `badge.tsx` — бейдж/«пилюля». Заменяет ad-hoc `inline-flex … rounded-full
  border … bg-muted/40 …` на лендинге (hero-пилюля «Платформа для…») и в
  `tournament-hero` («Активный турнир»), а также роль пользователя в списках
  админки.
- `select.tsx` — выпадающий список (Radix Select). Заменяет сырой `<select>` в
  `tournament-settings-form` (тип контакта). Клиентский компонент — форма и так
  `"use client"`.
- `skeleton.tsx` — плейсхолдер загрузки. Заменяет разнородный текст «Загрузка…»
  в списках админки на единый скелетон.

### 2. Конфиг навигации — `shared/config/site-config.ts`

- Починить/убрать пункты навигации, ведущие в никуда: пункт «Турниры»
  (`#tournaments`) — либо привести к существующему якорю секции турнира
  (`#tournament`), либо убрать, если секция не самостоятельна. Оставить только
  якоря, для которых на странице есть цель (`#features` при сохранении секции).
- Итог: `navItems` содержит только рабочие ссылки.

### 3. Роуты — `src/app/**`

- `page.tsx` (главная/лендинг):
  - Секция «Возможности» рекламирует несуществующие фичи (Участники, Судейство)
    → по FR-4 убрать преждевременную рекламу. Вариант: сократить до честного
    состояния (оставить только то, что есть) либо оформить как явный
    некликабельный роадмап с бейджем «скоро». По умолчанию — убрать
    dead-обещания; hero + профиль турнира остаются ядром страницы.
  - hero-пилюля → `Badge`; иконка-бокс фичи — упростить/через общий подход;
    сократить вложенность `div`.
  - Синхронизировать якоря с `site-config` (см. §2).
- `dashboard/page.tsx` — привести карточку профиля к единому виду; вынести
  повторяющуюся строку «label — value» в аккуратный компонент/паттерн, если
  дублируется (иначе оставить, но без лишних `div`).
- `(admin)/admin/page.tsx`, `(admin)/admin/create/page.tsx`,
  `(admin)/admin/tournament/page.tsx` — единый паттерн заголовка раздела и
  навигации «назад»: сейчас «назад» есть только в форме создания админа. Ввести
  консистентную навигацию по разделу (напр. общий header-паттерн admin-зоны или
  единая кнопка «назад»). Ссылки — через `next/link`.

### 4. Виджеты — `widgets/`

- `navbar/navbar.tsx` — унифицировать ссылки: логотип уже `Link`, а пункты
  навигации и ссылка «Админка» — сырые `<a>`. Внутренние переходы (`/admin`) —
  через `next/link`; якорные ссылки на секции остаются `<a href="#…">`, но
  только на существующие якоря (см. §2).
- `tournament-hero/tournament-hero.tsx` — пилюля «Активный турнир» → `Badge`;
  контакты-«пилюли» (ad-hoc `rounded-md border …`) → единый компонент-ссылка
  (напр. `Badge` c `asChild`/ссылкой или `Button variant="outline"`); заглушка
  «Турнир скоро появится» — согласовать с общим пустым состоянием. Чистая
  логика (`formatEventRange`, `contactHref`) — не трогаем, тесты сохраняем.

### 5. Фичи — `features/`

- `tournament-settings/ui/tournament-settings-form.tsx` — сырой `<select>` →
  `Select` (§1). Кнопки строк контактов и «+ Добавить» — согласовать варианты
  кнопок. Логика submit/маппинг — без изменений.
- `admin/ui/admin-list.tsx` — состояния загрузки → `Skeleton`; строка списка

  `features/admin/ui/` (напр. `user-row.tsx`); роль пользователя → `Badge`.
  Хуки/мутации (`useAdmins`, `usePromoteUser`…) — без изменений.
- `admin/ui/create-admin-form.tsx`, `auth/ui/auth-form.tsx` — единый паттерн
  состояния загрузки кнопки (сейчас «…» vs «Сохранение…»): ввести общий
  индикатор (спиннер/`Skeleton`-независимый) и единый текст. Поведение submit —
  без изменений.
- `auth/ui/auth-cta.tsx` — оставить как есть либо привести варианты кнопок к
  общему виду; функциональность (открытие диалога) неизменна.

### 6. Сущности — `entities/`

- `entities/user/ui/user-menu.tsx` — пустой файл-заглушка: удалить (FR-8).
  Реальный `UserMenu` живёт в `widgets/navbar/user-menu.tsx`.
- Проверить прочие пустые/неиспользуемые сегменты `ui/` и удалить.

### 7. Темы и токены — `src/app/globals.css`

- Палитру не переписываем. Допустимы точечные правки токенов на усмотрение
  реализации (§Принятые решения спеки, п.1), только через CSS-переменные
  `:root`/`.dark`. Проверить, что новые примитивы (`Badge`, `Select`,
  `Skeleton`) корректны в обеих темах (AC-6). Хардкод цветов в компонентах не
  вводить (FR-5).

- Server components vs client: новые примитивы — `Badge`/`Skeleton` серверные
  (без хуков), `Select` — клиентский (используется в client-форме). Правило
  «server component по умолчанию» сохраняется.
- State: изменений в state-менеджменте нет (server-state → RQ, UI-state →
  zustand) — рефактор презентационный.

## Тестирование

> См. ADR 0003 (стратегия) и ADR 0009 (TDD-цикл). Скриншотных тестов не вводим.

- Существующие Vitest-тесты остаются зелёными: `tournament-hero.test.ts`
  (`formatEventRange`, `contactHref`), `auth-dialog-store.test.ts`, fetchers
  (`requests.test.ts`), BFF `route.test.ts`/`route.e2e.test.ts`. Чистую логику
  этих модулей не меняем — тесты не должны «краснеть».
- `shared/config/site-config.ts` — если меняем форму `navItems`, покрыть

  инвариант «нет пунктов в никуда»), либо валидировать инвариант в тесте
  лендинга. Дешёвый unit на чистые данные.
- Новые чистые хелперы (если появятся при выносе паттернов) — покрыть Vitest.
- Компонентные/рендер-тесты презентационной верстки не добавляем (ADR 0003):
  корректность — через `tsc --noEmit`, `pnpm build`, ревью и ручную проверку в
  обеих темах.
- Прогон: `make test-web` (Vitest) + `pnpm exec tsc --noEmit` + `pnpm build`.

## Риски и открытые вопросы

- **Radix `Select`** — клиентский, порталы/поповеры: убедиться, что в форме
  внутри диалога/страницы корректно работает выбор, доступность и тема. Риск
  низкий (стандартный shadcn-компонент).
- **Пустое состояние страницы после удаления «Возможностей»** — если убрать
  секцию целиком, главная становится «hero + профиль турнира». Приемлемо для
  MVP; при желании — честный роадмап-блок (некликабельный, «скоро»). Решение —
  на этапе реализации, в рамках FR-4.
- **Регрессии поведения** — рефактор презентационный, но затрагивает много
  файлов. Митигируем: не трогаем логику/хуки/мутации, опираемся на существующие
  тесты (AC-7) и ручную проверку сценариев.
- **Объём** — правок много и они мелкие; риск «расползания» под видом полировки.
  Держим границу: только консистентность/чистка/dead-paths, без новых фич
  (Вне скоупа).
