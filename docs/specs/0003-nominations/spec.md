# Spec: Номинации турнира

> Артефакт SDD (ADR 0008). Описывает **ЧТО и ЗАЧЕМ**. Без технологий, API и
> схем БД — только предметная область. Реализация — в `plan.md`.

- Статус: done
- Дата: 2026-07-08

## Проблема

Турнир проводится не одним общим списком бойцов, а по **номинациям** —
отдельным дисциплинам/категориям (например «Лонгсворд про», «Сабля новички»,
«Меч-баклер»). У каждой номинации свои правила, планируемое число участников и
сопутствующие материалы (ссылка на регламент и прочее). Сейчас номинаций в
системе нет: организатор не может завести перечень дисциплин турнира, а
посетители не видят, по каким номинациям пройдут бои.

Нужно дать администратору возможность вести список номинаций активного турнира
(создавать, редактировать, удалять, упорядочивать), а посетителям — видеть этот
список на публичной странице турнира.

## Акторы

- **admin** — управляет номинациями активного турнира: создаёт, редактирует,
  удаляет, меняет порядок. Роль `ROLE_ADMIN` (RBAC уже есть).
- **гость / user** — видит список номинаций турнира на публичной странице. Не
  аутентифицирован или аутентифицирован как обычный пользователь; только чтение.

## Пользовательские сценарии

1. Как **admin**, я хочу создать номинацию с названием и описанием, чтобы
   заявить дисциплину, по которой пройдут бои.
2. Как **admin**, я хочу указать планируемое количество бойцов в номинации,
   чтобы обозначить её вместимость.
3. Как **admin**, я хочу приложить к номинации прочие данные (ссылку на правила
   и любую другую произвольную информацию), чтобы участники знали регламент.
4. Как **admin**, я хочу отредактировать или удалить номинацию позже, чтобы
   поддерживать перечень актуальным.
5. Как **admin**, я хочу задать порядок номинаций, чтобы они выводились в нужной
   последовательности.
6. Как **гость**, я хочу видеть список номинаций турнира с их описанием и
   деталями, чтобы понять, в чём смогу участвовать.

## Требования

- FR-1: Номинация **обязательно** принадлежит турниру: связь `tournament_id`
  указывается явно при создании и не может быть пустой. Клиент передаёт
  идентификатор турнира в запросах создания/списка/переупорядочивания; система
  проверяет, что такой турнир существует (в MVP, с единственным активным
  турниром, — что это активный турнир). Модель уже готова к мультитурнирности
  (номинация всегда привязана к конкретному турниру), без переписывания сущности.
- FR-2: Номинация включает поля: название (обязательное), описание, планируемое
  количество бойцов (опционально), прочие данные (произвольный набор
  ключ-значение), порядок в списке.
- FR-3: admin может создать номинацию активного турнира.
- FR-4: admin может изменить любое поле существующей номинации.
- FR-5: admin может удалить номинацию.
- FR-6: admin может задать порядок номинаций (переупорядочивание списка).
- FR-7: Список номинаций активного турнира и отдельная номинация доступны для
  чтения всем посетителям без аутентификации.
- FR-8: Управление номинациями (создание, изменение, удаление, порядок) доступно
  только роли `ROLE_ADMIN`.
- FR-9: Название номинации обязательно и непусто; в пределах одного турнира
  названия не повторяются (без учёта регистра).
- FR-10: Планируемое количество бойцов — целое неотрицательное число либо не
  задано (пустое). Оно информационное и в этой фиче ни с чем не связано
  (регистрации участников ещё нет).
- FR-11: Прочие данные — **типизированный набор необязательных полей**
  (`metadata`). В MVP это поле `rules_url` — ссылка на правила/регламент
  номинации. Все поля опциональны (могут быть не заданы). Схема **закрытая**:
  система принимает только объявленные поля; неизвестные ключи отклоняются.
  Набор полей расширяется в будущем (см. «Вне скоупа») без потери данных.
- FR-12: Пустые/незаданные поля номинации (описание, количество бойцов, прочие
  данные) не ломают отображение публичной страницы.

<Нефункциональные.>

- NFR-1: Чтение списка номинаций активного турнира — дешёвое (одна выборка по
  турниру), пригодно для SSR публичной страницы.
- NFR-2: Модель расширяема под мультитурнирность (номинация хранит связь с
  турниром по идентификатору).

## Критерии приёмки

- **AC-1** (создание админом)
  - Given: admin аутентифицирован
  - When: он создаёт номинацию с названием «Лонгсворд про», описанием и ссылкой
    на правила в прочих данных
  - Then: номинация сохраняется у активного турнира и появляется в списке
- **AC-2** (чтение публично)
  - Given: у активного турнира есть номинации
  - When: гость открывает публичную страницу турнира без токена
  - Then: он видит список номинаций с названием, описанием, количеством бойцов и
    прочими данными в заданном порядке
- **AC-3** (редактирование)
  - Given: существует номинация
  - When: admin меняет её описание и количество бойцов и сохраняет
  - Then: изменения сохраняются и отражаются в списке
- **AC-4** (удаление)
  - Given: существует номинация
  - When: admin удаляет её
  - Then: номинация исчезает из списка и перестаёт отдаваться при чтении
- **AC-5** (порядок)
  - Given: у турнира три номинации
  - When: admin задаёт новый порядок
  - Then: список (в т.ч. на публичной странице) отдаётся в этом порядке
- **AC-6** (защита записи)
  - Given: обычный пользователь (не admin) или гость
  - When: он пытается создать/изменить/удалить номинацию или сменить порядок
  - Then: запрос отклоняется (`Unauthenticated`/`PermissionDenied`)
- **AC-7** (обязательное название)
  - Given: admin создаёт номинацию
  - When: название пустое
  - Then: запрос отклоняется с ошибкой валидации, номинация не создаётся
- **AC-8** (уникальность названия)
  - Given: у турнира уже есть номинация «Сабля»
  - When: admin создаёт вторую номинацию «сабля» (в другом регистре)
  - Then: запрос отклоняется с ошибкой валидации/конфликта
- **AC-9** (неотрицательное количество бойцов)
  - Given: admin создаёт/редактирует номинацию
  - When: количество бойцов отрицательное
  - Then: запрос отклоняется с ошибкой валидации
- **AC-10** (частично заполненная номинация)
  - Given: у номинации задано только название
  - When: гость открывает публичную страницу
  - Then: номинация отображается названием, пустые блоки
    (количество/правила/прочее) не показываются и не ломают вёрстку
- **AC-11** (типизированные прочие данные)
  - Given: admin создаёт номинацию и указывает ссылку на правила
  - When: гость открывает публичную страницу
  - Then: `rules_url` отдаётся как типизированное поле и показывается ссылкой;
    неизвестные поля metadata при записи отклоняются с ошибкой валидации
- **AC-12** (обязательный турнир)
  - Given: admin создаёт номинацию
  - When: `tournament_id` не указан или указывает на несуществующий/неактивный
    турнир
  - Then: запрос отклоняется с ошибкой валидации/`NotFound`, номинация не
    создаётся

## Вне скоупа

- **Регистрация бойцов в номинации** — заявки участников, подсчёт фактического
  числа участников. Поле «количество бойцов» здесь чисто информационное.
- **Сетки, расписание, судейство по номинации** — отдельные фичи.
- **Мультитурнирность** — несколько турниров и выбор активного (см. спеку 0001);
  модель не мешает будущему расширению.
- **Расширение схемы прочих данных** — новые типизированные поля metadata сверх
  `rules_url` (напр. рулсет, требования к снаряжению, заметки) добавляются
  отдельными инкрементами. Строгая валидация формата (напр. проверка URL) — по
  мере надобности.
- **Загрузка файлов** (регламенты как вложения) — прочие данные хранят только
  ссылки/строки.
- Версионирование/история изменений номинаций.

## Принятые решения

Вопросы закрыты в обсуждении со стейкхолдером (2026-07-08):

1. **Границы модуля** — новый server-модуль `nomination` (bounded context со
   своей PG-схемой, ADR 0002). Связь с турниром — по `tournament_id`; жёсткого
   кросс-схемного FK нет (модульные границы). `tournament_id` **обязателен** и
   передаётся клиентом явно; сервер валидирует существование турнира через API
   модуля `tournament` (в MVP — что это активный турнир).
2. **Объём операций** — admin: полный CRUD + переупорядочивание; публично —
   чтение (список и одна номинация).
3. **Количество бойцов** — плановая вместимость: целое, опциональное, `>= 0`.
   Информационное, с регистрацией не связано.
4. **Прочие данные** — jsonb-поле `metadata` с **типизированной закрытой
   схемой** (набор необязательных полей). Хранится как jsonb (расширяемо без
   миграций), но контракт типизирован. В MVP одно поле — `rules_url` (ссылка на
   правила). Неизвестные ключи при записи отклоняются. В админке редактируется
   как типизированное поле формы.
5. **Название** — обязательное, уникальное в пределах турнира (без учёта
   регистра).
