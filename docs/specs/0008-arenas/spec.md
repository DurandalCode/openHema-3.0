# Spec: Площадки — ристалища/арены (arenas)

> Артефакт SDD (ADR 0008). Описывает **ЧТО и ЗАЧЕМ**. Без технологий, API и
> схем БД — только предметная область. Реализация — в `plan.md`.

- Статус: done
- Дата: 2026-07-13

## Проблема

На HEMA-турнире бои идут параллельно на нескольких физических **площадках**
(ристалищах / аренах). Чтобы ускорить сетку, оргкомитет запускает 2–5 ристалищ
одновременно, за каждым закреплён секретарь/судья, ведущий бои именно этой
площадки. Сейчас в системе понятия площадки нет — некуда «прикрепить» бой, негде
организовать рабочее место, ведущее бои конкретного ристалища.

Нужна базовая сущность **«площадка»** как:

- **элемент конфигурации турнира** — сколько ристалищ, как называются/нумеруются;
  админ ведёт этот список;
- **рабочее место / точка привязки боёв** — у площадки есть стабильный
  **админский URL** (страница управления площадкой), который позже наполнится
  управлением боями этой площадки.

В текущем инкременте — **намеренно минимальная** функциональность: админ **заводит,
правит и убирает (в архив)** площадки, а также открывает страницу управления
площадкой по стабильному URL. Сами бои и управление ими — **вне скоупа этого
домена**: площадка лишь предоставляет стабильную идентичность/URL, к которому бои
прикрепятся отдельной будущей фичей. Домен получается простым и независимым.

## Акторы

- **admin (оргкомитет/секретарь)** — создаёт, переименовывает/правит, архивирует
  и возвращает из архива площадки; управляет их порядком; открывает страницу
  управления площадкой. Роль `ROLE_ADMIN` (в MVP секретарь = admin, как в
  0005/0007).
- **судья/секретарь площадки** (будущее) — работает на админском URL конкретной
  площадки, ведёт её бои. **Вне скоупа** этого инкремента (пока = admin).

## Домен

- **Площадка (arena)** — именованное место проведения боёв в рамках турнира.
  Независимая сущность: **не ссылается** на бойцов, номинации, заявки. В будущем
  **бои будут ссылаться на площадку**, а не наоборот — поэтому площадка живёт
  своей жизнью и ничего не знает о других доменах.
- Каждая площадка имеет **стабильную идентичность** → стабильный админский URL
  страницы управления. Идентичность **не меняется** при переименовании и
  архивации, чтобы закладку/ссылку на «свою» площадку не сломать.

### Атрибуты

- **имя** — непустое (например, «Ристалище 1», «Главная арена»);
- **описание / локация** — свободный текст, необязательное (например, «у входа,
  ковёр 5×5»);
- **порядок** — позиция в списке площадок; управляется admin (ристалища принято
  нумеровать/упорядочивать);
- **статус** — `активна` (по умолчанию) | `в архиве`.

### Статус и архивация

- **Архивация — это «удаление» этого домена**: физического удаления нет. Убрать
  площадку = перевести в архив; она остаётся в системе. Операция **обратима**
  (вернуть из архива). Причина, по аналогии с обратимым выводом бойца (0007):
  на площадку в будущем будут ссылаться бои — историю «где проходил бой» терять
  нельзя, поэтому строки не удаляем физически уже сейчас.
- Архивная площадка не участвует в активной конфигурации (позже — не принимает
  новые бои), но её страница по стабильному URL остаётся доступной и помечена
  «в архиве».

## Пользовательские сценарии

1. Как **admin**, я хочу добавить площадку (имя, при желании описание), чтобы
   отразить реальные ристалища турнира и подготовить место для будущих боёв.
2. Как **admin**, я хочу видеть список площадок турнира в заданном порядке, чтобы
   понимать текущую конфигурацию арен.
3. Как **admin**, я хочу переименовать/поправить площадку (имя, описание), чтобы
   поддерживать список в порядке.
4. Как **admin**, я хочу управлять порядком площадок в списке, чтобы отразить
   нумерацию ристалищ.
5. Как **admin**, я хочу убрать лишнюю/ошибочную площадку в архив и при
   необходимости вернуть её, не теряя её идентичность.
6. Как **admin**, я хочу открыть страницу управления конкретной площадкой по
   стабильному URL (пока — каркас, бои появятся позже), чтобы позже вести на ней
   бои с этого рабочего места.

## Требования

Функциональные:

- FR-1: Площадка — самостоятельный bounded context (своя PG-схема, ADR 0002),
  независимый от других доменов. В этом инкременте площадка ни на кого не
  ссылается, и другие домены на неё не ссылаются.
- FR-2: Атрибуты площадки: **имя** (непустое), **описание/локация** (свободный
  текст, необязательное), **порядок** (позиция в списке), **статус** (активна /
  в архиве). Модель расширяема под будущую привязку боёв и назначение судей.
- FR-3: admin может **создать** площадку с непустым именем (описание — опционально).
- FR-4: admin может **переименовать/поправить** площадку (имя, описание). Пустое
  имя отклоняется.
- FR-5: admin может **архивировать** площадку («убрать»); физического удаления
  нет. Операция **обратима** — admin может **вернуть из архива**.
- FR-6: admin видит **список площадок** турнира, отсортированный по заданному
  порядку; активные и архивные различимы.
- FR-7: admin может **менять порядок** площадок в списке.
- FR-8: У площадки есть **стабильный идентификатор** для админского URL страницы
  управления; идентификатор **не меняется** при переименовании и архивации.
- FR-9: Существует **страница управления конкретной площадкой** по её стабильному
  URL. В этом инкременте — **каркас** (реквизиты площадки + место под будущее
  управление боями); бои наполнят её отдельной фичей.
- FR-10: Все операции над площадками — только `ROLE_ADMIN`. Публичного чтения в
  этом инкременте нет (домен админский).

Нефункциональные:

- NFR-1: Модель расширяема под будущую привязку боёв, назначение судей и
  публичную навигацию «где мой бой» — без ломки существующих данных.
- NFR-2: Чтение списка площадок — дешёвое (пригодно для админского экрана и
  будущего SSR, если появится публичный вид).

## Критерии приёмки

- **AC-1** (создание)
  - Given: admin
  - When: создаёт площадку с непустым именем (и, опционально, описанием)
  - Then: площадка появляется в списке (со статусом «активна») и доступна по
    своему стабильному URL
- **AC-2** (пустое имя отклоняется)
  - Given: admin
  - When: пытается создать или переименовать площадку в пустое имя
  - Then: операция отклонена с ошибкой валидации
- **AC-3** (правка)
  - Given: существующая площадка
  - When: admin меняет её имя и/или описание
  - Then: изменения видны в списке и на странице площадки; её стабильный URL не
    изменился
- **AC-4** (порядок)
  - Given: несколько площадок
  - When: admin меняет их порядок
  - Then: список отображается в новом порядке
- **AC-5** (архивация)
  - Given: активная площадка
  - When: admin убирает её (в архив)
  - Then: площадка помечена «в архиве», остаётся в системе, физически не удалена;
    её URL остаётся доступным
- **AC-6** (возврат из архива)
  - Given: архивная площадка
  - When: admin возвращает её
  - Then: площадка снова активна с прежней идентичностью и URL
- **AC-7** (стабильный URL)
  - Given: существующая площадка
  - When: admin открывает её страницу управления по URL и затем переименовывает
    и архивирует её
  - Then: страница остаётся доступной по тому же URL (идентификатор не менялся)
- **AC-8** (права)
  - Given: обычный пользователь или гость
  - When: пытается создать / изменить / архивировать / вернуть / переупорядочить
    площадку
  - Then: операция отклонена (`Unauthenticated` / `PermissionDenied`)

## Вне скоупа

- **Бои**: их создание, привязка к площадке, управление боями на странице
  площадки — отдельная будущая фича. Здесь — только каркас страницы и стабильный
  URL.
- **Физическое удаление** площадки — сознательно нет; убираем архивацией
  (обратимо), чтобы не терять будущую историю боёв.
- **Расписание/тайминг**, распределение боёв по площадкам, назначение судей.
- **Публичное чтение** списка площадок и навигация «где мой бой» для
  зрителей/бойцов — появится вместе с боями/расписанием.
- **Роль «судья/секретарь площадки»** и доступ к отдельной площадке без полного
  admin — секретарь = admin (как в 0005/0007).
- **Уникальность имён/номеров** площадок системой не форсится — забота admin.
- **Мультитурнирность** — площадки принадлежат одному турниру; выбор турнира вне
  скоупа (как в 0007).

## Принятые решения

Зафиксировано со стейкхолдером (2026-07-13):

1. **Площадка (arena) — независимый bounded context**, ни на кого не ссылается;
   бои (будущее) будут ссылаться на неё, а не наоборот.
2. **Скоуп операций: создать + править (rename/описание) + убрать + вернуть +
   переупорядочить.** «Убрать» = **архивация (обратимо)**, физического удаления
   нет — по аналогии с обратимым выводом бойца (0007), чтобы не терять будущую
   историю боёв.
3. **Атрибуты:** имя (обяз., непустое), описание/локация (необяз., свободный
   текст), порядок (admin управляет), статус (активна / в архиве).
4. **Стабильный идентификатор → стабильный админский URL** страницы управления
   площадкой; не меняется при переименовании и архивации.
5. **Страницу управления площадкой делаем уже сейчас как каркас** (реквизиты +
   место под будущее управление боями); бои наполнят её позже.
6. **Домен админский** — публичного чтения в этом инкременте нет; появится вместе
   с боями/расписанием.
7. **Уникальность имён/номеров не форсим** — это ответственность admin.
8. **Английское имя сущности/модуля — `arena`** (папка `0008-arenas`).
9. **Мультитурнирность вне скоупа** — площадки принадлежат одному турниру.
